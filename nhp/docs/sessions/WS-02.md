## Goal
Land the NHP domain as **rubix collection records** (`kind:"collection"`) — tenant, site, gateway,
network, meter, meter-type, register — with typed fields, `required`/`unique` rules, relation links,
and write-rules (incl. the per-network device limit). Resolve the closed-enum strategy. After this
WS, creating an NHP record validates against its shape; no later WS has to define the schema.

## Dependencies

{Text, Number, Bool, Date, File, Relation}`, gate write-path validation (fail-open / strict), and
`required`/`unique` — verified in [../OVERVIEW.md](../OVERVIEW.md).

2. **Enum strategy — NHP-only, NO rubix change.** Model every closed enum as a `text` field and

   - the collection `writeRule` (an access/validation expression — *data*, not core code) rejecting

   - the UI dropdown (WS-04/05) constraining input.

   `rubix/crates/rubix-core/src/collection/field.rs` is explicitly OUT OF SCOPE — NHP runs on the

3. Add the device-limit `writeRule` on `meter`: reject a create whose parent `network` already has
   `max_devices` meters (defence-in-depth behind the wizard's check). If write-rules can't express a
   count query, log to TODOs.md and rely on the wizard check for the POC (note the gap).
4. Verify validation end-to-end: a well-formed record of each kind writes; a bad one (missing
   required, bad enum, dup unique) is rejected with a clear error.

## Outcome (2026-06-16)

**How the 7 collections are registered.** As `kind:"collection"` records POSTed to
the unmodified rubix `POST /records` HTTP surface
(`rubix/crates/rubix-server/src/http/records/create.rs`). No NHP Rust, no rubix
change. The definitions are data — one file per collection under
[`nhp/collections/`](../../collections/): `tenant.mjs`, `site.mjs`, `gateway.mjs`,
`network.mjs`, `meter.mjs`, `meter-type.mjs`, `register.mjs` (barrel
`definitions.mjs`, dependency order). `register-collections.mjs` upserts them
idempotently (skips a collection whose `name` already exists). Auth is the
service-account header pair (`x-rubix-subject`/`x-rubix-secret`,
`rubix/crates/rubix-server/src/auth.rs`); against a `--seed-dev` server in
namespace `acme` that is `acme_operator`/`operator-demo` (the seeded operator
holds the `ingest-publish` record-write capability — there is no HTTP path to
create principals, so the registrar relies on a seeded one). `make collections` /
`make collections-check` wrap the two scripts.

**Enum + uniqueness + device-limit enforcement decision — verified, then chosen.**
Before committing to writeRule-based enforcement I re-grepped the gate validate
path. **rubix does NOT evaluate any `writeRule`**: `CollectionDef`
(`rubix/crates/rubix-core/src/collection/def.rs`) parses only `name` + typed
`schema`; its docstring states `writeRule`/`listRule` are "preserved as raw JSON"
because "their evaluation surface is a separate open question … modelling them now
would imply an enforcement path that does not exist yet." The gate's validate step
(`rubix/crates/rubix-gate/src/command/validate.rs`) calls only
`CollectionDef::validate`, which checks **`required` + field TYPE only**, and
explicitly **does not check `unique`**
(`rubix/crates/rubix-core/src/collection/validate.rs`). Confirmed live over HTTP:
a missing-required and a wrong-type record are rejected `422`; a duplicate `unique`
value and an out-of-set enum value are **admitted** by the gate.

So enforcement splits, honestly:

| Rule | Where it runs |
| --- | --- |
| `required` present | **rubix gate** (native, `422`) |
| field type (text/number/bool/date/relation) | **rubix gate** (native, `422`) |
| closed enum allowed-set (`net_type`, `protocol`, `status`, register `fn_code`/`datatype`/`byte_order`/`chart_type`) | **NHP client layer** ([`enforce.mjs`](../../collections/enforce.mjs)) + UI dropdown (WS-04/05) |
| `unique` | **NHP client layer** (the gate does not enforce it) |
| per-network device limit (`max_devices`) | **NHP client layer** + wizard (WS-06); the gate cannot express a count query |

This is the kickoff's prescribed POC outcome: client-side enforcement now, native
enforcement logged as a RUBIX-TEAM item. No rubix patch, no invented enforcement.
Enums are `text` fields; the allowed sets live in
[`enums.mjs`](../../collections/enums.mjs) as the single source of truth.

**JSON fields.** rubix has no `json` FieldType. `network.params`,
`meter-type.registers`, and `register.alarm` (DOMAIN-MODEL) are therefore **left
out of the declared schema** — rubix admits undeclared content fields as raw JSON
(`collection/validate.rs` "Undeclared fields are allowed"), so these pass through
unvalidated, which is the intended POC behaviour. Noted in each module.

**Verification.** `make collections` twice (created → exists, idempotent) then
`make collections-check`: one VALID record per kind writes (7×`200`); one INVALID
per kind is refused (gate `422` for missing-required on tenant/site/meter-type/
register; client rejection for bad enum on gateway/network/meter); plus a
dup-unique reject and a device-limit reject, both via the client layer. All
checks passed. `git status --short rubix/` is empty.
