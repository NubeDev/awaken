## Goal
A one-command seed that populates a realistic NHP portfolio through the gate, plus a stand-in for
the (not-built) polling service: fake `status`/`last_seen` and back-filled `history` for
`history=true` registers, so every UI workstream has live-looking data to render.

## Dependencies
WS-02 (collections committed). Modelled on rubix's seed
(`rubix/crates/rubix-server/src/seed/{portfolio,history,cast}.rs`) — same gate-written pattern.

## Standard
[rubix/docs/FILE-LAYOUT.md](../../../rubix/docs/FILE-LAYOUT.md). Verb/spec-per-file: portfolio
builder, history generator, status faker as separate files. No one mega seed file.

## Current state (re-verify first)
- rubix seed: `--seed-dev` arg in `rubix/crates/rubix-server/src/main.rs` calls `seed_dev`; the
  portfolio is built record-by-record **through the gate** so it carries real audit/undo/events.
  Reuse `seed/history.rs`'s series generator for the back-fill.
- Seed shape to build is specified in [../SEED.md](../SEED.md).

## Tasks (incremental — commit per task, `WS-03:` prefix)
1. **Meter-types:** 1–2 full types with real Modbus register maps (3-phase: V L1/L2/L3, A L1/L2/L3,
   kW, kWh, Hz, PF) — each register with address/datatype/scale/byte_order/unit/quantity/history/
   chart_type/chart_group/alarm per [../DOMAIN-MODEL.md](../DOMAIN-MODEL.md).
2. **Portfolio:** 2 tenants × 2 sites; per site 1–2 gateways, each with a mix of 485 + ethernet
   networks (with `max_devices` caps), and several meters stamped from the types. Apply the standard
   tags (`tenant:…`, `site:…`, `gateway:…`, `meter:…`, `group:…`, `quantity:…`) so dashboards
   auto-build (DASHBOARDS.md).
3. **Users/teams:** admin + operator + viewer per tenant, and a **poller service-account** principal
   with write grants — per [../ADMIN.md](../ADMIN.md) roles.
4. **Fake the poller:** write plausible `status`/`last_seen`; back-fill history for `history=true`
   registers (reuse rubix `seed/history.rs`); a ticker for live values during `make dev SEED=1`.
   This is the ONLY place NHP fabricates device data — it stands in for the external service.
5. Wire `SEED=1` into `nhp/Makefile` (mirror rubix's `SEED_FLAG`), off by default.

## Done gate
- `make -C nhp dev SEED=1` populates the portfolio and the seed run exits clean.
- A scripted check (or query console) confirms expected counts: 2 tenants, 4 sites, ≥1 meter-type
  with a full register set, meters tagged, history present for `history=true` registers.
- Seed is idempotent-ish (repopulates on boot, documented), off by default.

## Outcome (2026-06-16)

**Re-verified contracts first.** Against the live tree:
- `--seed-dev` cast (`rubix/crates/rubix-server/src/seed/cast.rs`) provisions, **per tenant
  namespace** (`acme`, `globex`): `{ns}_admin/admin-demo` (Role::Admin, `IngestPublish`),
  `{ns}_operator/operator-demo` (Operator, `IngestPublish`+`ExternalQuery`),
  `{ns}_viewer/viewer-demo` (Viewer, no grants), `{ns}_analyst/analyst-demo`, and
  `{ns}_agent/agent-demo` (PrincipalKind::**Extension**, Operator, `RuleInvoke`). Confirmed live in
  the boot log. Subjects use an **underscore** separator. The seed authenticates as
  `acme_operator`/`operator-demo` (reused from WS-02).
- `POST /records` → `RecordDto { id, namespace, content, tags, created, updated }`
  (`rubix/crates/rubix-server/src/dto/record.rs`); the new record's id is the relation target a child
  links to. Relations are id strings; the gate does not verify the target exists.
- **No HTTP tag-attach route.** Tags are graph edges projected onto `RecordDto.tags` on the **list**
  read only (`rubix/crates/rubix-server/src/http/records/`); single reads return none, and there is
  no route to *write* a tag edge. So NHP carries its standard tags in **`content.tags`** (a content
  array), written over the same records API. WS-07 reads `content.tags`. Documented in `seed/tags.mjs`.
- History is just records: a `kind:"history"` record with `register`/`ts`/`value` in content
  (mirrors rubix's `kind:"reading"`, `seed/history.rs`), each row carrying its own `ts` because the
  gate stamps `created = now`.

**Principal-seeding decision — REUSE the `--seed-dev` cast (no HTTP provisioning needed).** WS-02
noted "no HTTP path to create principals" because the *operator* is not an Admin. Re-checked:
`POST /principals` (`rubix/crates/rubix-server/src/http/admin/principals.rs`) DOES exist and works,
but `require_admin` (`http/admin/guard.rs`) needs Role::Admin in the namespace — so it would have to
run as `acme_admin`/`admin-demo`. **It is unnecessary**: `--seed-dev` already provisions every role
WS-03 asks for, per tenant — **admin** (`{ns}_admin`), **operator** (`{ns}_operator`), **viewer**
(`{ns}_viewer`), and a **service-account** for the poller (`{ns}_agent`, PrincipalKind::Extension
with write-class grants). The NHP seed therefore creates **no principals**; it reuses the seeded
cast and documents the credential map. This is a clean reuse, not a limitation — NHP adds zero rubix
surface. (If a future WS needs *extra* NHP-specific principals, the route to use is
`POST /principals` authed as `{ns}_admin`; logged below as a follow-up, not built here.)

**What the seed creates (verified counts, `seed/check.mjs` all-pass).** 2 tenants (`acme`, `globex`)
× 2 sites = 4 sites; 5 gateways; 7 networks (a mix of `485` + `ethernet`, each with a `max_devices`
cap + net-type `params`); 14 meters stamped from the meter-types (each carries `meter_type` +
`meter_type_version`, DOMAIN-MODEL §versioning); 105 register records stamped from the type
register-defs; **2 meter-types** — `acme-pm5560` with the **full 3-phase register set** (V L1/L2/L3,
A L1/L2/L3, kW, kWh, Hz, PF — 10 registers, each with full Modbus + presentation + alarm metadata)
and a compact `acme-em24`. 4368 history rows back-filled (48h hourly per `history=true` register).

**Tags — one shared module.** `seed/tags.mjs` is the single source of truth for the standard tags
(`tenant:` `site:` `gateway:` `network:` `meter:` `group:<chart_group>` `quantity:<q>`
`meter-type:<key>`), imported by the portfolio builder now and by WS-06/WS-07 later so producer and
consumer never drift. Each record carries its hierarchy tags (+ register grouping/quantity tags) in
`content.tags`.

**Faked poller.** `seed/poller-status.mjs` bakes deterministic `status`/`last_seen` onto every
gateway + meter at create time (most online, ~1-in-7 offline so the site-rollup has a degraded case,
offline devices get a stale `last_seen`). `seed/history.mjs` generates a deterministic 48h hourly
wave per `history=true` register (oscillation per quantity, cumulative ramp for energy) — the NHP
stand-in for the external polling service. No live-value ticker was added (optional/nice-to-have;
the baked-in snapshot + history is enough for every UI WS to render). History is **not** re-written
on a re-seed (only freshly-stamped registers back-fill), so re-runs don't duplicate the series.

**Seed flow + Makefile.** `SEED=1` stays the rubix `--seed-dev` switch (it provisions the namespace +
principals the NHP seed authenticates as — a *prerequisite*, not the NHP data). The NHP portfolio is
a separate `make seed` (alias `make seed-nhp`) that runs the collections registrar then the portfolio
builder against the running backend; `make seed-check` asserts the done-gate counts. Required order
documented in the Makefile: `make dev SEED=1` → `make seed` → `make seed-check`. Off by default.

**Files** (FILE-LAYOUT: one responsibility each, all ≤~250 lines): `seed/tags.mjs` (shared tags),
`seed/meter-types.mjs` (templates), `seed/portfolio-plan.mjs` (the portfolio shape, data),
`seed/portfolio.mjs` (the builder), `seed/history.mjs` (series generator), `seed/poller-status.mjs`
(status faker), `seed/seed.mjs` (orchestrator entrypoint), `seed/check.mjs` (count verification).
Reuses `nhp/collections/client.mjs` (HTTP) + the registrar; no rubix change.

**Verification.** Booted `rubix --seed-dev` on `:8094`, `make seed` → all records written, `make
seed-check` → all 15 assertions PASS, re-ran `make seed` → 0 new records/history (idempotent-ish),
`git status --short rubix/` empty.

## Notes / follow-ups
- Write through the gate as the tenant operator — do NOT bypass with direct store writes (keeps
  audit/undo/events real, which the admin/dashboard WSs rely on). **Done** — everything is a
  `POST /records` as `acme_operator`.
- **Single namespace for the POC.** The seed authenticates as `acme_operator` and writes all NHP
  records into the `acme` namespace; the two NHP *tenants* (`acme`, `globex`) are **records** with
  `tenant:<key>` tags, not separate rubix namespaces. True per-tenant namespace isolation (writing
  `globex`'s records as `globex_operator`) is a multi-tenant-onboarding concern for WS-05/06 — noted,
  not needed for the POC's data-to-render goal.
- **`listRecords('site')` returns 6, not 4** — rubix's own `--seed-dev` also writes 2 `kind:"site"`
  records in the `acme` namespace (its Haystack-ish demo). NHP's 4 sites are distinctly keyed/tagged;
  the overlap is harmless (the check uses `≥`). A future cleanup could namespace-prefix or filter, but
  it doesn't affect NHP rendering.
- Tags live in `content.tags`, not as rubix graph edges (no HTTP tag-write route). If a later WS wants
  true tag-graph edges, that needs a rubix tag-attach HTTP route — a **RUBIX-TEAM** item (logged in
  TODOs.md), not an NHP edit. For the POC `content.tags` is sufficient and WS-07 reads it.
