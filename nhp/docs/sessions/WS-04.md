## Goal
The back-of-house surface for **meter-types and their Modbus register maps**: create/edit/clone a
meter-type, edit each register's protocol metadata + semantics + presentation + alarms, set
`history` on/off, choose `chart_type` and `chart_group`, and handle meter-type **versioning**
(bump on edit; stamp + re-sync on meters). This is the heart of [../ADMIN.md](../ADMIN.md) §1–4.

## Dependencies
WS-02 (collections + enum strategy), WS-03 (data to show), WS-01 (app shell).

## Standard
[rubix/docs/FILE-LAYOUT.md](../../../rubix/docs/FILE-LAYOUT.md). One file per verb/route/panel —
e.g. `meter-types/list.tsx`, `meter-types/edit.tsx`, `meter-types/register-row.tsx`. Reuse
rubix-old/ui's `features/admin` + data-table + form scaffolding.

## Current state (re-verify first)
- rubix-old/ui has `features/admin`, `components/data-table`, form/drawer primitives — the parts bin.
- Register fields + alarm shape + versioning rules: [../DOMAIN-MODEL.md](../DOMAIN-MODEL.md)
  (`register` table, §Alarms, §versioning).

## Tasks (incremental — commit per task, `WS-04:` prefix)
1. **Meter-type list + CRUD:** list, create, clone-as-new, delete (data-table + drawer/form).
2. **Register-map editor:** a table editor over `registers[]` — per row all fields from DOMAIN-MODEL
   (`address`/`fn_code`/`datatype`/`word_count`/`byte_order`/`scale`/`offset`/`signed`, then
   `unit`/`quantity`/`history`/`chart_type`/`chart_group`/`precision`, then `alarm`). Enum fields use
   the WS-02 enum strategy. Bulk-paste (CSV/JSON) import for big maps.
3. **Alarms:** edit the `alarm` threshold ramp per register (reuse the FieldConfig threshold model,
   [rubix DASHBOARDS-SCOPE §7](../../../rubix/docs/design/DASHBOARDS-SCOPE.md)).
4. **Versioning:** every save bumps `version`; existing meters untouched. Show how many deployed
   meters are on older versions; a per-meter "re-apply type" that diffs the meter's registers vs the
   current type version and applies on confirm (stamp + `meter_type_version`). All via the gate.

## Done gate
- `pnpm -C nhp/ui build` green, no TS errors.
- Manually (or via a smoke script): create a meter-type, add registers with enums/units/history/
  alarms, save (version bumps), edit again (bump again, existing seeded meters unchanged), re-apply
  to one meter (its registers update, `meter_type_version` advances).

**Feature (`nhp/ui/src/features/meter-types/`, one responsibility per file, all ≤~250 lines):**
- `list.tsx` — meter-type table: key/name/manufacturer/version/register-count + **deployed rollup**
  (total meters, "N out of date" badge when `meter_type_version < version`). Row actions: edit, clone,
  re-apply, delete.
- `edit.tsx` — full-page meter-type editor (metadata + register table). **Create** writes version 1;
  **edit** bumps `version` (DOMAIN-MODEL §versioning) and never touches meters; **clone** seeds the form
  from a source type with a fresh key + version 1.
- `register-table.tsx` / `register-row.tsx` — the register-map editor: a row per `registers[]` entry with
  **every** DOMAIN-MODEL field editable inline (address/fn_code/datatype/word_count/byte_order/scale/
  offset/signed, unit/quantity/history/chart_type/chart_group/precision, alarm). Enum fields are
  dropdowns from the shared options. Add-row + bulk import.
- `alarm-editor.tsx` — the `{thresholds:[{value,severity}], for}` ramp editor, expanded inline per row.
- `bulk-import.ts` + `bulk-import-dialog.ts[x]` — paste a CSV (header row of field names) or JSON array;
  appended to the register set.
- `reapply-dialog.tsx` + `versioning.ts` — per-meter re-apply: pick an (out-of-date-first) meter, show
  the register **diff** (added/removed/changed/unchanged) of its stamped registers vs the type's current
  `registers[]`, and on confirm re-stamp + advance `meter_type_version`.
- `hooks.ts` — React Query hooks (list meter-types/meters/registers; create/update/delete; re-apply).
- `register-defaults.ts` — blank-row factory.

**How it writes to rubix.** Everything is the generic records API (`nhp/ui/src/api/records.ts`, typed):
`POST /records {content}`, `GET /records?kind=…`, `PATCH /records/:id {content}`, `DELETE /records/:id`
(bare routes, NOT `/api/v1`). Meter-types are `kind:"meter-type"`; meters `kind:"meter"`; registers
`kind:"register"` keyed `${meterKey}--${defKey}` (matches WS-03's stamping, verified live). Re-apply
deletes the meter's old register records, creates the new stamped set, then patches the meter's version
— all gate-audited writes. Auth: `api/client.ts` now sends the `x-rubix-subject`/`x-rubix-secret`
service-account pair from `VITE_RUBIX_SUBJECT`/`VITE_RUBIX_SECRET` (the seeded `acme_operator`) when no
Bearer token is set — same auth the WS-02/03 scripts use; documented in `.env.example`.

**Enum sharing with WS-02 (no duplication).** `nhp/collections/enums.mjs` stays the Node source of truth.
The UI cannot import it into the tsc build (it lives outside `src` and is untyped), so `enums/options.ts`
is the TS mirror the UI consumes, and `enums/options.unit.test.ts` imports BOTH (Vite resolves the `.mjs`
for the test) and asserts every set is identical — so any drift fails the test gate. A small
`enums/collections-enums.d.ts` types the `.mjs` import so no `@ts-ignore` is needed. `options.ts` is
reusable by WS-05 (net_type/protocol/status) and WS-06.
