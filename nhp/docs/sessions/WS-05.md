## Goal
The infrastructure + identity admin: CRUD for gateways and their networks (485/ethernet, protocol,
per-network `max_devices` device limit), and user/team/role management on the rubix principal +
grant surface. Covers [../ADMIN.md](../ADMIN.md) §5–6.

## Dependencies
WS-02 (collections), WS-03 (data), WS-01 (shell). Independent of WS-04 (different collections) but
runs after it per the one-branch sequential rule.

## Standard
[rubix/docs/FILE-LAYOUT.md](../../../rubix/docs/FILE-LAYOUT.md). One file per verb/route. Reuse
rubix-old/ui `features/admin/{members,teams,access}.tsx` for users/teams/roles.

## Current state (re-verified 2026-06-16, file:line)
- rubix admin API is built: routes in `rubix/crates/rubix-server/src/http/admin/mod.rs:33`:
  `POST/GET /principals`, `GET/PATCH/DELETE /principals/:subject`,
  `GET /principals/:subject/grants`, `PUT/DELETE /principals/:subject/grants/:capability`.
  Bare routes (NOT `/api/v1`), same family as `/records`.
- **All principal/grant routes require `Role::Admin`** via `require_admin`
  (`http/admin/guard.rs:25`, called at the top of every handler in `principals.rs`/`grants.rs`).
  The records-API client uses the seeded **operator** (`acme_operator`), which is NOT admin —
  so user CRUD needs the seeded **admin** credential (`acme_admin` / `admin-demo`,
  `rubix/crates/rubix-server/src/seed/cast.rs:36-42`). See the reachability decision below.
- Wire shapes (`rubix/crates/rubix-server/src/dto/admin.rs`):
  - `CreatePrincipalRequest { subject, kind, role, secret? }` -> `201 CreatedPrincipalDto { subject,
    namespace, kind, role, secret? }` (secret echoed once only when the server minted it).
  - `PrincipalDto { subject, namespace, kind, role }`; `UpdatePrincipalRequest { role }` (role change only).
  - `kind in {user, extension}` (`parse_kind:198`), `role in {viewer, operator, admin}` (`parse_role:210`).
  - **Subject is namespace-local** on the wire: send `admin`, server stores `acme_admin` and strips
    the `{namespace}_` prefix from every response (`prefix_subject`/`strip_subject_prefix`).
  - `GrantDto { subject, namespace, capability }`; grants are `PUT/DELETE .../grants/:capability`.
- Guards: cannot demote/delete the **last admin** (`409`, `principals.rs:179`).
- The poller service account is `acme_agent` (kind `extension`, role operator;
  `cast.rs:70-76`) -- surfaced read-only in the UI.
- NHP role set (viewer/operator/admin/root) is in [../ADMIN.md](../ADMIN.md) §6 (root is the
  cross-tenant onboarding path, not this in-namespace surface).

## Tasks (incremental — commit per task, `WS-05:` prefix)
1. **Gateways:** list/create/edit/delete (`gateway` collection); show online/offline + `last_seen`
   (poller-written, read-only here).
2. **Networks:** under a gateway, CRUD `network` records — `net_type` (485/ethernet, switches the
   `params` form: baud/parity vs ip/port), `protocol`, `max_devices`. Show device count vs cap.
   Unlimited networks per gateway.
3. **Device-limit enforcement:** the create-network/add-meter UI shows remaining capacity and blocks
   over-cap; rely on WS-02's `writeRule` as the backend backstop (or note its gap if write-rules
   can't count).
4. **Users & teams:** CRUD principals, assign NHP **roles** (viewer/operator/admin) and team
   membership via `/principals` + grants. Surface the poller service-account as a managed principal.

**Gateways + networks (`features/gateways/`, one responsibility per file, all ≤~250 lines):**
- `list.tsx` — gateway table with poller-written **status + last_seen shown READ-ONLY**
  (`status-badge.tsx`); each row expands to its networks (`network-table.tsx`). Create/edit/delete via
  `gateway-form.tsx`.
- `gateway-form.tsx` — key/name/**site**/model/host. `site` is a **required** relation (the gate
  enforces it — verified live: a create without `site` returns `422 ... site: required field is
  missing`), so the form has a site picker over `kind:"site"` records (relation by record id, matching
  the WS-03 seed). NEVER sets `status`/`last_seen`; on edit it spreads existing content first so the
  poller's values survive the PATCH.
- `network-table.tsx` / `network-form.tsx` — CRUD `kind:"network"` under a gateway (gateway relation by
  record id). `net_type` (485/ethernet) switches the params sub-form: `params-485.tsx`
  (baud/parity/stop/data-bits) vs `params-ethernet.tsx` (ip/port). `protocol` + `max_devices` common;
  net_type/protocol dropdowns come from the shared `enums/options.ts` (no re-listing). Networks per
  gateway unlimited.
- `capacity.ts` + `capacity-badge.tsx` — **device-limit (client-side).** `capacityFor(network, meters)`
  computes used/cap/remaining (the TS mirror of `enforce.mjs`'s `deviceLimitViolation` concept, which
  is Node-only). The badge shows `used/cap` (amber when full); the network edit form BLOCKS lowering
  `max_devices` below the meters already on the network. The add-meter over-cap block proper lives in
  WS-06's add-meter wizard (this WS owns plain network CRUD, not meter add) — `capacity.ts` is the
  reusable primitive for it. rubix's gate can't count (WS-02 finding, already logged RUBIX-TEAM in
  TODOs.md), so this is defence-in-one for the POC, as briefed.

**Users & roles (`features/users/`) + the principal-CRUD reachability decision.**
- Wired to the REAL rubix admin API (`api/admin.ts`, NOT mocked): `GET/POST /principals`,
  `PATCH/DELETE /principals/:subject`, `GET /principals/:subject/grants`.
- `list.tsx` lists every principal (users + the `extension` service accounts, e.g. the poller `agent`,
  flagged as a service account); `role-select.tsx` inline-changes the NHP role (viewer/operator/admin);
  `edit.tsx` creates a principal (subject/kind/role, optional secret — blank ⇒ server mints one,
  surfaced ONCE in the success toast by `hooks.ts`); `grants.tsx` shows a principal's capability grants
  READ-ONLY.
- **REACHABILITY DECISION.** All `/principals*` routes require `Role::Admin` (`http/admin/guard.rs`),
  and the records-API service account is the seeded **operator** — which is NOT admin (verified live:
  operator on `GET /principals` → **403**). So the Users admin authenticates as the seeded
  **`acme_admin` / `admin-demo`** via a new per-request `auth` override on `api/client.ts.request`
  (admin creds are `VITE_RUBIX_ADMIN_SUBJECT`/`_SECRET`, defaulted to the seed and documented in
  `.env.example`). This is the "simplest honest thing" the brief asked for: real routes, real admin
  auth, documented cred — nothing faked, no rubix patch. Subjects are namespace-LOCAL on the wire
  (UI works in `alice`, server stores `acme_alice`).
- **Grant MUTATION (PUT/DELETE …/grants/:capability) is intentionally NOT exposed** in the POC UI
  (display only): which capabilities a role needs is backend policy and the seed already confers the
  right grants per role (`cast.rs`). Honest, reachable, not faked — granting is left to the seed / a
  later WS. **Teams**: rubix has no team/group primitive on the principal surface (only
  role-in-namespace + per-principal grants); ADMIN.md §6 frames "teams" as a grant-grouping concept
  with no backend object today, so there is nothing to CRUD — roles are the membership model in the
  POC. Noted, not stubbed.
