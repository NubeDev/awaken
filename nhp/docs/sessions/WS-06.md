## Goal
Guided multi-step wizards that author many gate-audited records (and their tags) at once, applying
NHP conventions so dashboards auto-build. Covers [../WIZARDS.md](../WIZARDS.md): new tenant, new
site, new gateway **with N networks** ("30 networks"), bulk meters, new user/team, and the combined
"add everything" flow.

## Dependencies
WS-04 + WS-05 (the create paths and the device-limit rule the wizards orchestrate), WS-02/03/01.

## Standard
[rubix/docs/FILE-LAYOUT.md](../../../rubix/docs/FILE-LAYOUT.md). One file per wizard, one file per
step within a wizard (`gateway-wizard/{gateway-step,networks-step,preview-step}.tsx`). No mega
wizard file.

## Current state (re-verify first)
- Wizard flows + behaviours specified in [../WIZARDS.md](../WIZARDS.md). Each step is an ordinary
  collection write (WS-04/05 paths). rubix-old/ui form/stepper primitives are the parts bin.

## Tasks (incremental — commit per task, `WS-06:` prefix)
1. **Wizard shell:** a reusable stepper with a final **preview** of records-to-create, then a
   batched write with per-record results; resumable on partial failure (don't restart from zero).
2. **Gateway + N networks** (the headline): one step generates N network records from a count, a
   `net_type`/`protocol`/`max_devices`, a naming pattern (`gw-01-net-{n}`), and per-type `params`
   defaults. Preview all N, confirm, write. Test with N=30.
3. **Bulk meters onto a network:** pick network (shows remaining = `max_devices` − current), pick a
   meter-type, add a range of bus addresses; **block** over-cap; stamp registers + tags per meter.
4. **Tenant / site / user wizards:** tenant (root, fresh namespace via `/tenants` + first admin),
   site (name/address/timezone), user/team (role assignment).
5. **Combined flow:** chain tenant→site→gateway(+networks)→meters→users, threading parent ids, one
   final preview of the whole tree.

## Done gate
- `pnpm -C nhp/ui build` green, no TS errors.
- Smoke: run the gateway+30-networks wizard → 30 networks created + tagged; run bulk-meters →
  capped correctly, meters stamped; run the combined flow on a fresh tenant → dashboards (WS-07)
  show the new tree.

**Wizard shell (`features/wizards/_shared/`, task 1).**
- `batch-write.ts` — the batched, RESUMABLE writer. A wizard produces a `PlannedRecord[]` (each a
  records-API `content` + label + optional `parentRefs` for late-bound parent ids — a child planned
  before its parent's record id exists). `runBatch` writes in ORDER, resolving parentRefs from
  already-written ids, returns a per-record `BatchResult`, and on re-run SKIPS already-`ok` records
  (resume, not restart — WIZARDS.md §Principles). Each write is `createRecord` (gate-audited); nothing
  faked.
- `stepper.tsx` — the reusable `WizardShell`: numbered step rail, per-step `valid` gating Back/Next, a
  final PREVIEW phase (`buildPlan()` → all records-to-create, nothing written yet), then RUN with live
  per-record status and a Retry that re-runs only the failed records.
- `preview-table.tsx` — the preview/result table (kind, label, the standard tags, post-run status).
- `hooks.ts` (read-only list hooks the wizards thread parents from) + `ancestry.ts` (walks a network's
  relation chain up to its tenant KEYS for tag context).

**Gateway + N networks — THE HEADLINE (task 2).** `gateway-wizard/{plan.ts, gateway-step.tsx,
networks-step.tsx, gateway-wizard.tsx}`. `plan.buildGatewayPlan` emits 1 gateway + N networks from
count + net_type/protocol/max_devices + a `gw-01-net-{n}` pattern + per-type params defaults; each
network's `gateway` relation is a `parentRef` to the gateway's yet-to-be-written id. Reuses the
485/ethernet params sub-forms. `plan.unit.test.ts` proves N=30: 31 records, distinct keys 1..30, every
network late-binds the gateway, shared tags applied, net params carried.

**Bulk meters + cap-block (task 3).** `meters-wizard/{plan.ts, meters-wizard.tsx}`. Pick a network
(the dropdown + a banner show remaining = `max_devices` − current via `capacity.ts`), pick a meter-type,
give an inclusive bus-address range + key/name patterns. THE CAP-BLOCK: the address step is invalid
(`valid: requested >= 1 && !overCap`) when `requested > capacity.remaining`, so Preview is disabled and
a red message explains — reusing WS-05's `capacityFor`. `buildMetersPlan` stamps, per address, a meter
(with `meter_type` + `meter_type_version`) and its registers from the type's `registers[]`, keyed
`${meterKey}--${defKey}` with registerTags (adds group:/quantity:) — the SAME stamping the seed/WS-04
do; registers late-bind their `meter` relation. `plan.unit.test.ts` covers it.

**Tenant / site / user (task 4).** `tenant-wizard`, `site-wizard`, `user-wizard`.
- **TENANT DECISION (confirmed from WS-03 §Notes):** in the POC NHP tenants are RECORDS
  (`kind:"tenant"`, tagged) in the single `acme` namespace — NOT separate rubix namespaces. So the
  tenant wizard writes one `kind:"tenant"` record (key/name, `namespace:"acme"`, tags []), mirroring the
  seed. The WIZARDS.md §1 fresh-namespace + first-admin steps are OUT of POC scope (no namespace-
  provisioning path) — adding principals is the user wizard.
- Site: name/address/timezone + a parent-tenant relation picker; siteTags from the tenant key.
- User: principals are NOT records — the user wizard reuses WS-05's admin-authed `useCreatePrincipal`
  (`/principals`), not the records batch. "Teams" have no backend object in the POC (WS-05 reachability),
  so role is the membership model.

**Combined "add everything" (task 5).** `combined-wizard/{plan.ts, combined-wizard.tsx}`. Five steps

the whole greenfield tree with EVERY parent relation late-bound (all records are new), one final preview
of the tree before writing. Meters land on the first generated network (cap-checked vs that network's
`max_devices`). `plan.unit.test.ts` proves ordering (parents before children), parentRef threading
across all 6 levels, and the shared tags at every level.
