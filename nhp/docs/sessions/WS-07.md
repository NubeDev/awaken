## Goal
Auto-built dashboard pages driven by the tag graph: a page per hierarchy level (tenant/site/gateway/
meter) generated from tags, with chart grouping (all voltages in one chart), online/offline rollups,
and alarms. Covers [../DASHBOARDS.md](../DASHBOARDS.md), using rubix's chart/query machinery
([rubix DASHBOARDS-SCOPE](../../../rubix/docs/design/DASHBOARDS-SCOPE.md)).

## Dependencies
WS-02 (tags/shapes), WS-03 (history + status to render), WS-01 (shell). Best after WS-06 so the
combined wizard's output has somewhere to land, but only hard-needs WS-02/03.

## Standard
[rubix/docs/FILE-LAYOUT.md](../../../rubix/docs/FILE-LAYOUT.md). Pure renderer discipline: a widget
is a pure function of `{ widget, data }`; one place fetches (DASHBOARDS-SCOPE §7/§8). One file per
widget builder, one per auto-build level.

## Current state (re-verify first)
- rubix/ui has the Laminar chart-builder + dashboards components and `api/{boards,charts}.ts`;
  `POST /query` + `POST /query/batch` are built (OVERVIEW). recharts is the base; gauge/heatmap on a
  lazy ECharts island (DASHBOARDS-SCOPE §8).
- Auto-build rules (tags → pages), grouping, status rollup, alarm panel: [../DASHBOARDS.md](../DASHBOARDS.md).

## Tasks (incremental — commit per task, `WS-07:` prefix)
1. **Query client + batch:** wire one batched query per board (`/query/batch`), visibility-aware
   refresh, UTC windows + relative tokens, site-local time (DASHBOARDS-SCOPE §3/§5/§6).
2. **Auto-build:** walk the tag graph to generate a board per level (tenant cards → site cards →
   gateway networks/limits → meter charts). Deterministic: same tags → same page. Allow a saved
   hand-tuned board to override a generated one for a scope.
3. **Chart grouping:** render one multi-series chart per `group:<g>` (e.g. all voltages); cross-meter
   `quantity:<q>` panels on site pages.
4. **Status + alarms:** online/offline tiles + hierarchy rollup from poller-written `status`/
   `last_seen`; live updates via `/ws/records`; an alarm panel + per-chart threshold colouring from
   register `alarm` thresholds; operator acknowledge (gate-audited).
5. **Widgets:** the common tier (line/bar/area/pie/stat/status/table) on recharts; gauge/heatmap only
   if a seeded register needs them, on the lazy ECharts island.

## Outcome (2026-06-16)

**Re-verified "Current state" first — and corrected a stale contract.** The OVERVIEW table and this
spec both claim `POST /query/batch` exists; it does NOT on the frozen rubix tree. `rubix/crates/
rubix-server/src/http/query/mod.rs` mounts ONLY `POST /query` (a single read-only SQL statement,
gated on the `external-query` capability — `run.rs`), with no `/query/batch` route and no batch DTO
(`dto/query.rs` is `{sql}`→`{rows}` only). recharts 3.8.1 IS in `nhp/ui/package.json` (confirmed).
enums/tags.ts is intact (the WS-06 parity-guarded mirror). `git status --short rubix/` empty at start.
Logged the doc drift as a follow-up below; did NOT edit rubix.

**Query mechanism — POC fallback to `/records` (documented, blessed by WS-07).** Because batch isn't
there, the dashboards read the seeded data over the already-wired, operator-authorised `/records` API
(`api/records.ts`) and do windowing/grouping/aggregation CLIENT-SIDE — the spec's explicit "fall back
to /records + a simple history query + a refetch timer" path. `query/batch.ts` is the ONE fetcher
(React Query hooks per kind + one whole-`history` read, shared by every page — the §7 "one place
fetches" discipline). The pure auto-build + widget layers are unchanged if a real batch lands later;
only that file swaps. A KEY detail re-confirmed live: NHP's standard tags live in `content.tags`
(WS-03 — rubix has no HTTP tag-attach route), so the server-side `?tag=` filter (which joins the
graph projection) does NOT see them; the builders filter on `content.tags` via `auto-build/scope.ts`
using the SAME tag builders from `enums/tags.ts` (tenantTag/siteTag/gatewayTag/networkTag/quantityTag)
the seed + wizards write — so a page keys off exactly the tags the data carries.

**Live/refresh — visibility-aware timer, NOT `/ws/records` (documented POC simplification).** The
spec blesses a visibility-aware refetch as "sufficient for the POC if WS is fiddly". `query/use-refresh.ts`
pauses the timer on a hidden tab and catches up on return; `use-dashboard-refetch.ts` re-runs the
`['dash']` queries on the chosen interval (Off · 30s · 1m · 5m). A `/ws/records` live subscription
(WS client + per-principal row filter + debounced invalidation) is the later upgrade behind the same
hook; out of the POC timebox.

**How tags drive auto-build (deterministic — same tags ⇒ same page).** Four pure level-builders, one
file each, walk `content.tags`:
- `tenant-board.ts` — `tenant:<key>` → site cards; each rolls up its gateways' status (rollup.ts:
  "degraded" if any offline), counts meters under `site:<key>`, sums active alarms (alarm-eval.ts).
- `site-board.ts` — `site:<key>` → gateway cards + a cross-meter `quantity:power` panel (one series
  per meter — DASHBOARDS.md §"Chart grouping" cross-cut).
- `gateway-board.ts` — `gateway:<key>` → status+last_seen + a network table (device count per
  `network:<key>` vs `max_devices`).
- `meter-board.ts` — ONE multi-series trend per `chart_group` (all voltages together), stat tiles for
  no-`history` registers, an alarm panel, status+last_seen. Join: register `content.meter`=meter id,
  `content.key`=`<meterKey>--<defKey>`; history `register`=bare `<defKey>` — matched on meter id + suffix.

**Widgets (recharts only; NO ECharts island).** `widgets/{line,bar,area}.tsx` on recharts (the easy
tier, DASHBOARDS-SCOPE §8), `stat.tsx`/`table.tsx`/`status-tile.tsx`/`alarm-panel.tsx` are DOM tiles,
`empty.tsx` is the never-fabricate-a-zero state. Each is a PURE function of `{widget,data}` (§7); the
container is the only thing that fetches. No seeded register uses `gauge`/`heatmap`, so per §8 the lazy
ECharts island is OMITTED — the dashboards chunk ships recharts only (no `echarts`/`echarts-for-react`
dep; the grep "echarts" hits are all the substring inside "**r**echarts"). Threshold colouring: a chart
draws warning/critical reference lines from the register alarm ramp (`field-config.ts` thresholdLines),
the SAME ramp the alarm panel + stat tiles colour by (`severityFor`) — "what you see is what alarms".

**Live-verified against the seed** (`rubix-server --seed-dev` on :8094 + `make seed`, data dir inside
gitignored `rubix/rubix-data`, removed after): 2 tenants / 6 sites / 5 gateways / 7 networks / 14
meters / 105 registers / 4368 history rows. Walked the real pipeline (curl + a verification script):
tenant→acme has 2 sites; `acme-plant` rolls up to **degraded** (gateways online+offline); a meter
exposes `power`/`energy`/`power_factor` chart_groups; a register joins **48** history points; the vite
proxy serves `/records` through to :8094 and the route is in `routeTree.gen.ts`. Build: `pnpm -C nhp/ui
build` GREEN, no TS errors, no @ts-ignore. Tests: 55/55 unit (added field-config/rollup/meter-board
specs incl. a threshold-crossing case). The Playwright BROWSER suite needs a browser binary not present
here — pre-existing infra, unrelated to WS-07 (unit suite + build cover the gate).

**Alarm note (honest).** The seeded voltages oscillate ~224–236 V; the seeded ramp warns at 250 /
crits at 253, so NO alarm fires live on the current seed — the alarm PANEL is correctly empty and 0
alarms roll up onto the cards. The threshold ramp still COLOURS the chart (the warning/critical
reference lines render at 250/253 regardless of a crossing), and the crossing logic (severityFor /
buildMeterBoard / alarmCountsByMeter) is proven by unit tests with an over-threshold sample. Nudging
the seed to cross a threshold is WS-03's domain, not WS-07's; left as-is. A live-firing demo would seed
one register near its limit — noted for WS-08 polish.
