## Barcode scheme (the round-trip contract)
- The meter-type **`key`** is already a stable, unique identifier
  (DOMAIN-MODEL ¬ßmeter-type: `key` is `required, unique`). The barcode encodes that
  key, namespaced with a fixed prefix: **`nhp-mt:<key>`** (e.g. `nhp-mt:acme-pm5560`).
- The prefix namespaces NHP meter-type codes so a stray scanned QR (a shipping label,
  a URL) does NOT accidentally resolve to a meter-type ‚Äî decode returns `null`.
- **Encode** = `nhp-mt:${key}`. **Decode** = strip the prefix ‚Üí the key.
  **Resolve** = find the meter-type whose `content.key === key`.
- Stored as a `text` `barcode` field on the meter-type, set to the derived value at
  create/clone (the admin can override). It is shown read-only on edit (it tracks the
  key for a stable graphic) ‚Äî see edit UI.
- **Backfill-safe.** Existing meter-types (acme-pm5560, acme-em24) have no stored
  `barcode`. The resolver derives the SAME code from `key` when `barcode` is absent,
  so they resolve with no data migration. The encode/decode round-trip is the single
  module `enums/barcode.ts`, drift-guarded by `barcode.unit.test.ts`.

## Library choices
- **`qrcode`** (render): generate a QR data-URL/canvas for the printable label. QR
  over a 1D barcode because phone cameras decode QR far more reliably (brief's
  preference). Tiny, well-maintained, zero peer deps.
- **`@zxing/browser` + `@zxing/library`** (scan): camera-based QR decode in the
  browser, with a still-image decode path for the manual/upload fallback. The de-facto
  browser barcode lib.
- Encode/decode of the *scheme* (not the graphic) stays in `enums/barcode.ts`; QR
  rendering in `features/meter-types/barcode-label.tsx`; camera scanning in
  `features/wizards/scan-wizard/scanner.tsx`. One concept per file (FILE-LAYOUT.md).

**How the scan resolves a code ‚Üí meter-type.** The barcode encodes the meter-type's
already-unique `key` behind the `nhp-mt:` prefix. `resolveBarcode` strips the prefix
and finds the meter-type whose `content.key` matches ‚Äî so a backfilled seed type (no
stored `barcode`) resolves identically to a freshly-created one. A code without the
prefix returns `null` (unknown-barcode error), so a stray shipping-label/URL scan never
mis-resolves.

**Capacity + tags.** Capacity reuses `capacity.ts` (`capacityFor`): the network picker
disables full networks and the step is invalid when `remaining < 1`. Tags come ONLY
from `enums/tags.ts` (`meterTags`/`registerTags`) ‚Äî the same builders the seed and other
wizards use, so a scan-added meter carries the exact tag chain the dashboards auto-build
on. Tags are written into `content.tags` (NHP off-loads to content; rubix has no HTTP
tag-attach route) ‚Äî the established convention.

**Assumptions / deviations.**
- **Barcode = QR of `nhp-mt:<key>`**, chosen over a 1D barcode (phone-camera
  reliability, brief's stated preference) and over a generated random code (deriving
  from the already-unique `key` makes existing types backfill-safe with NO migration).
- **No seed backfill needed.** Because the code derives from `key`, the seed's
  meter-types resolve without a stored `barcode` ‚Äî the resolver handles it. The editor
  DOES persist a `barcode` on the next save, but it is never required.
- **One meter per scan** (scan-to-add is a single-device flow); bulk stays in the
  meters-wizard. The plan builder mirrors `meters-wizard/plan.ts` for one address.
- `@zxing/library` pinned to `0.22.0` to satisfy `@zxing/browser@0.2.0`'s peer range
  (the latest 0.23 tripped a peer warning); no behavioural impact.
- The two `react-hooks/exhaustive-deps` WARNINGS in scan-wizard.tsx are the SAME pattern
  the existing meters-wizard.tsx carries (derived list feeding a `useMemo`); kept for
  consistency ‚Äî warnings, not errors, and the gate is build + tests.
