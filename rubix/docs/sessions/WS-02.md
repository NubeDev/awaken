# WS-02 — Generic record model + tag graph

Status: Done
Started: 2026-06-14T15:20:13Z
Finished: 2026-06-14T15:41:00Z

## Goal
Deliver the schemaless document record and the `record→tagged→tag` graph that
replaces a fixed schema (SCOPE principle 4: generic, not domain-specific). The
platform processes data + tags, not a baked-in equipment/site/point ontology;
structure comes from tagging on the graph. This is the data substrate every
read, rule, and query above it operates over.

## Dependencies
WS-01 (committed, `8d2a135a`). Consumes the `rubix-core` error enum / `Id` type.

## What landed
`rubix-core` gains a `record` and a `tag` module, verb-per-file, all <400 lines.

- `record/`: `Record { id, namespace, content (serde_json::Value), created,
  updated }` plus `create_record` / `read_record` / `update_record` /
  `delete_record`. A private `row::RecordRow` maps the domain `Id` (plain
  string) to/from SurrealDB's reserved `id` `RecordId` so the domain type stays
  free of store identity concerns. `update_record` MERGEs only `content` +
  `updated` so `created` is preserved. `delete_record` clears outbound `tagged`
  edges in the same statement batch (SurrealDB does not cascade edge deletes).
- `tag/`: `Tag { id, name }` plus `create_tag` / `delete_tag` / `attach_tag`
  (idempotent `RELATE record->tagged->tag`) / `detach_tag` /
  `find_records_by_tags` (Haystack-style set intersection over the graph:
  `array::len(array::intersect(->tagged.out, $tags)) = $wanted`). `delete_tag`
  clears inbound `tagged` edges.

## Assumptions / deviations
- **Layering.** The spec places CRUD verbs in `rubix-core`. `rubix-core` sits
  below `rubix-store`, so it cannot depend on `StoreHandle`. The verbs therefore
  take a `&surrealdb::Surreal<Db>` connection (exactly what `StoreHandle::raw()`
  yields), and `rubix-core` takes a direct `surrealdb` dependency. Dependency
  direction stays clean: core never depends on store.
- **Table existence.** SurrealDB 3.x errors on `SELECT` from a missing table, so
  `rubix-store::init_schema` (the WS-01 seam built for exactly this) now declares
  `record` / `tag` / `tagged` as `SCHEMALESS` (`tagged` `TYPE RELATION`),
  idempotent via `IF NOT EXISTS`. This is a minimal additive change to the
  existing seam, not a new contract. The rubix-core integration fixture
  (`tests/db/open.rs`) mirrors the same definitions because core cannot depend on
  the store crate.
- **Nested test layout.** The spec's `tests/record/<verb>_test.rs` layout is not
  auto-discovered by Cargo, so each is registered as an explicit `[[test]]`
  target in `rubix-core/Cargo.toml`. The shared `open_memory_db` fixture lives in
  `tests/db/open.rs` and is `#[path]`-included (a subdir, not a test binary).

- Unit: `Record`/`Tag` construction + persisted-row round-trip.
- Integration (kv-mem): record CRUD round-trip; create→read; update bumps

  record clears its `tagged` edges; empty tag set matches nothing.
