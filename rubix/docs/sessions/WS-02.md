## Goal

Give suspended (awaiting-approval) agent runs an operator surface. STATUS.md AI-layer: HITL
escalation already returns a `SuspendTicket` and `POST /agent/chat` surfaces `status:
awaiting_approval` + `run_id`, but **the resume endpoint (operator approve/cancel) needs the
persistent-backend run store** — which does not exist yet. Inbound dispatch runs are likewise
"fire-and-log — no run registry/operator surface for in-flight dispatched runs."

This WS ships that store + the resume/cancel HTTP surface.

## Scope / deliverables

- `runs` table + store methods: persist a run on creation (chat or dispatch), update status
  (running/suspended/completed/cancelled), store the `SuspendTicket` for suspended runs.
- HTTP: `GET /api/v1/runs` (list, filter by status), `GET /api/v1/runs/{id}`,
  `POST /api/v1/runs/{id}/resume` (operator approve → re-drive the gated write),
  `POST /api/v1/runs/{id}/cancel`. Thin handlers → one domain fn each.
- Resume re-applies the suspended write through the priority array with gating re-checked.
- OpenAPI registration for the new routes + DTOs.
- Integration tests: suspend a run via the escalation band, list it, resume it (write lands), cancel
  another (store untouched).

## What landed

Five commits on `rubix-gaps`:

1. `792f5443` — persist agent runs with held HITL writes: `runs` table + index
   (`store/schema.rs`), store methods `create_run` / `list_runs(status)` /
   `get_run` / `settle_suspended_run` (atomic one-shot, returns the held write
   then clears it; 409 if not suspended) (`store/runs.rs`), and the domain model
   `RunRecord` / `RunStatus` / `RunOrigin` / `PendingWrite` (`agent/run_record.rs`).
2. `f52d41ec` — chat and dispatch persist their runs: `api/agent/chat.rs` and the
   `dispatch` module both drive activations through `run_and_persist`; `store`
   threaded through `Dispatcher::launch` → `run_dispatch` → `dispatch_spark`.
3. `b2160582` — operator surface: `GET /api/v1/runs` (status filter),
   `GET /api/v1/runs/{id}`, `POST /api/v1/runs/{id}/resume` (re-applies the held
   write through the priority array, gating re-checked; 409 on a settled run),
   `POST /api/v1/runs/{id}/cancel` (204, store untouched); OpenAPI registration.
4. `92f2d912` — release the blocked loop: `SuspendCaptureSink` records the run's
   events and signals on `RunFinish { Suspended }`; `run_and_persist` races the
   run task against that signal and `cancel_by_run_id`s the loop on suspend, then
   recovers the held write from the suspended `write_point` `ToolCallDone` event.

The full-local backend blocks the agent loop after suspending (it waits for an
operator decision on a live channel that this architecture never feeds). rubix
owns run persistence instead: the operator surface re-applies the held write
directly, so on the suspend signal the loop is cancelled to release it and the
`suspended` `RunRecord` is the durable source of truth.

This also resolves the pre-existing `chat_reports_awaiting_approval_when_a_write_suspends`
hang (logged in TODOs.md by WS-01): `cargo test --workspace` now terminates,
125 tests green, clippy clean.
