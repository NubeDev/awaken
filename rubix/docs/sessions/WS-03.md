## Goal

History is SQLite-only today. STACK-DEISGN.md specifies a Parquet history store via `object_store`
(edge + cloud tiers) and a `his` DataFusion `TableProvider` over Parquet partitions. Ship the
Parquet write path + a `TableProvider` so `/query` and `/his/rollup` can read tiered history.

## Scope / deliverables

- A Parquet partition layout (by site/point/time bucket) written via `object_store` (local FS tier
  for edge, gated to keep the SQLite path as the live/recent tier).
- A `his` `TableProvider` that unions Parquet partitions (cold) with the SQLite recent tier (hot),
  so existing `/query` and `/his/rollup` keep working unchanged across the tier boundary.
- A rollover/flush path moving aged SQLite history rows into Parquet partitions.
- Env toggle for the Parquet tier (e.g. `RUBIX_HIS_PARQUET` + a path/store URL).
- Tests: write history ‚Üí flush to Parquet ‚Üí query reads across both tiers and rollup aggregates are
  correct over the union.

## Execution log

Two commits, one per logical section.

1. `feat(query): two-tier his` ‚Äî the query-side cold tier in `rubix-query`:
   - `his/schema.rs`: the canonical `his` Arrow schema (`point_id`, `ts`, `value`,
     all Utf8), shared by both tiers and the union provider.
   - `his/partition.rs`: partition addressing ‚Äî `his/point=<id>/date=<YYYY-MM-DD>/<flush>.parquet`.
   - `his/write.rs` (`write_partitions`, `HisRow`): group rows by (point, UTC day),
     encode each group as a Parquet file via `ArrowWriter`, `put` on the object store.
   - `his/read.rs`: list every partition under the root prefix, decode into batches.
   - `his/hot.rs`: read the SQLite `his` rows into a batch over the shared schema.
   - `his/table.rs` (`HisTable`): a `TableProvider` whose `scan` unions the SQLite
     hot batch with all Parquet cold batches into one `MemTable`.
   - `his/tier.rs` (`HisTier`): a local-filesystem `object_store` handle backing both
     read and flush; `QueryEngine::with_his_tier` registers `HisTable` for `his`.
   - Tests (`tests/his_tier.rs`): query unions hot+cold, rollup aggregates across the
     boundary, no-tier stays SQLite-only, empty cold tier reads hot-only.

2. `feat(server): flush aged his` ‚Äî the write/retention boundary in `rubix-server`:
   - `store/his_flush.rs` (`Store::his_aged`/`his_delete_aged`): read rows older than
     a cutoff as `HisRow`s (rendering the UUID-blob `point_id` to its canonical string,
     matching the query provider), and delete them after the cold-tier write.
   - `his/flush.rs` (`flush_aged`): read ‚Üí write Parquet ‚Üí delete, so a failure before
     the delete leaves the hot tier intact (rows simply re-flush next time).
   - `api/his/flush.rs` (`POST /api/v1/his/flush`): operator/scheduled trigger with an
     optional `older_than` cutoff; 503 when no tier is configured.
   - `main.rs`: `RUBIX_HIS_PARQUET` opens the local tier, attaches it to the query
     engine and `AppState.his_tier`.
   - Tests (`api_tests/his.rs`): flush ages rows into Parquet, the hot tier drains, and
     the union `/query` still returns them; flush without a tier is 503.

## Assumptions / notes

- The cold tier is a local-filesystem `object_store` (edge tier). The read/write/flush
  code is generic over `ObjectStore`, so a cloud/remote store is a `HisTier` constructor
  change only ‚Äî no logic change. Cloud tiering / replication is out of WS-03 scope.
- `value` is carried as its stored JSON text on both tiers (cast to a number only at
  rollup time), so a row round-trips through Parquet byte-identically.
- The flush is operator/scheduler-triggered (a stored board can call `/his/flush`); no
  background retention timer is added ‚Äî that belongs with retention-policy config, not
  this gap.

## Follow-ups (not blockers)

- Partition pruning: `HisTable::scan` currently reads every partition then lets DataFusion
  filter. Point/date predicate pushdown to skip partition files would help large cold tiers.
- A scheduled retention trigger (interval board or a dedicated timer) once a retention
  policy/window is specified.
