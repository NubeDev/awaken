# WS-03 — Identity + scoped read session (the read enforcement point)

## Goal
Deliver one principal model for users **and** extensions (SCOPE principle 5:
everything is a scoped principal; extensions are service accounts in SurrealDB
auth) and the **read enforcement point**: reads run on a gate-issued scoped
SurrealDB session enforced by SurrealDB's own row-level permissions, never
proxied per message. This is the load-bearing split (contract #1) that makes
native reads/live-queries cheap while staying scoped.

## Dependencies
WS-01 (store handle, error enum, the scoped-session issuance seam in
`rubix-store::issue_session`) and WS-02 (records to scope reads over). Fills the
WS-01 session seam with the real issuance + permission wiring.

## Scope / deliverables
- Crate(s): `rubix-gate`.
- A `Principal` type: subject, namespace/tenant, kind (`user | extension`), role.
- `authenticate`: token → `Principal`.
- `issue_session`: mint a **scoped SurrealDB session** bound to the principal's
  namespace (signs in as the principal's service account / scope, not root).
- SurrealDB table **row-level permissions** keyed on the session principal, so
  the engine — not the app — decides which records the session may read.
- A read path that runs SELECTs on the scoped session (records from WS-02),
  returning only the principal's namespace data.
- Load-bearing contracts honored: #1 (reads → gate-issued scoped SurrealDB
  session, not per-message proxy) and #2 (data-record perms are SurrealDB-native
  via the scoped session, distinct from the app-enforced capability layer).

## Tests
- Unit (`#[cfg(test)]`): `authenticate` maps a valid token to the right
  `Principal` (subject/namespace/kind/role) and rejects an invalid token;
  `Principal` kind covers both `user` and `extension`.
- Integration (`tests/`, SurrealDB kv-mem): `issue_session` returns a session
  scoped to the principal's namespace; a scoped read over WS-02 records returns
  only that namespace's records; a principal in namespace A reading a record in
  namespace B gets empty / denied — proving enforcement is SurrealDB-native, not
  an app filter; the same identity model authenticates a `user` and an
  `extension` principal.
