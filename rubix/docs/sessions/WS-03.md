- Design decision (SurrealDB v3.1.4, embedded engine): a scoped session is a
  **clone** of the store's `Surreal<Db>` (a fresh `session_id` over the same
  `Arc`'d datastore) that has been `signin`'d through a **record access method**
  (`DEFINE ACCESS principal ON DATABASE TYPE RECORD`). Signin binds `$auth` to
  the principal record; the `record` table's `PERMISSIONS FOR select WHERE
  namespace = $auth.namespace` then enforces row-level reads natively. The root
  store handle session is unaffected. This satisfies contract #1 (reads run on a
  gate-issued scoped SurrealDB session, not a per-message proxy) and contract #2
  (data-record perms are SurrealDB-native, distinct from the app capability
  layer). Verified against `surrealdb-core` `iam::signin::db_access` (sets
  `session.au` to a non-owner Auth so permissions enforce) and the SDK
  `Surreal::clone` session model.

## Deviations from the proposed file layout
- `Principal` lives in `rubix-core` (`crates/rubix-core/src/principal/`), not
  `rubix-gate`, to honor the crate map contract. The gate keeps authenticate /
  session / permission / read. WS-04 keys off the same `rubix_core::Principal`.
- Added `rubix-gate::provision_principal` (owner-side identity registration) and
  `rubix-core::list_records` — both required for the read flow; each is a
  single-responsibility verb.
- `DEFINE TABLE OVERWRITE record … PERMISSIONS` layers read scope onto WS-01's
  `record` table; root (store-handle) sessions bypass permissions so WS-02 CRUD
  is unaffected. Scoped sessions get `create/update/delete NONE` — the command
  path (WS-05) writes through the root handle, not the scoped session.
- SurrealDB v3 renamed `type::thing` → `type::record`; the `SIGNIN` query uses
  `type::record('principal:' + $subject)`.

## Goal
Deliver one principal model for users **and** extensions (SCOPE principle 5:
everything is a scoped principal; extensions are service accounts in SurrealDB
auth) and the **read enforcement point**: reads run on a gate-issued scoped
SurrealDB session enforced by SurrealDB's own row-level permissions, never
proxied per message. This is the load-bearing split (contract #1) that makes
native reads/live-queries cheap while staying scoped.

## Dependencies
WS-01 (store handle, error enum, the scoped-session issuance seam in
`rubix-store::issue_session`) and WS-02 (records to scope reads over). Fills the
WS-01 session seam with the real issuance + permission wiring.

## Scope / deliverables
- Crate(s): `rubix-gate`.
- A `Principal` type: subject, namespace/tenant, kind (`user | extension`), role.
- `authenticate`: token → `Principal`.
- `issue_session`: mint a **scoped SurrealDB session** bound to the principal's
  namespace (signs in as the principal's service account / scope, not root).
- SurrealDB table **row-level permissions** keyed on the session principal, so
  the engine — not the app — decides which records the session may read.
- A read path that runs SELECTs on the scoped session (records from WS-02),
  returning only the principal's namespace data.
- Load-bearing contracts honored: #1 (reads → gate-issued scoped SurrealDB
  session, not per-message proxy) and #2 (data-record perms are SurrealDB-native
  via the scoped session, distinct from the app-enforced capability layer).
