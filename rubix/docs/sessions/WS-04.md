## Assumptions / deviations
- The no-escalation rule is "only an Admin may administer grants, and only in
  the grantee's own namespace". The spec said "a principal cannot grant itself,
  or another, a capability it lacks the authority to grant"; `Role::Admin` is the
  coarse authority band SCOPE defines, so admin-in-namespace is the fail-closed
  reading. A finer "grantor must itself hold the capability + a grant-delegation
  capability" model can layer on later without changing this contract.
- `Capability` does not derive serde (rubix-gate has no serde dep); the stable
  storage/wire form is the explicit kebab-case string. DTO/serde wiring is WS-16.
- `capability/kind.rs`'s string parser is named `parse` not `from_str` to avoid
  the clippy `should_implement_trait` lint.

## Goal
Deliver the **second authz layer**: app-enforced capability grants for
non-record / cross-plane actions — register a datasource, invoke a rule, publish
ingest, query an external source, subscribe a Zenoh key-space. These are
explicitly **not** SurrealDB row-level perms (that is WS-03's layer); both
enforcers key off the *same* principal. This is the honest "one identity, two
enforcement layers" of SCOPE, and it must fail closed.

## Dependencies
WS-03 (committed). Consumes the `Principal` type and the scoped read/write path
so grants are persisted and read within the caller's own scope. Builds on the
`rubix-gate` crate.

## Scope / deliverables
- Crate(s): `rubix-gate` (new `capability` module).
- A `Capability` enum (datasource-register, rule-invoke, ingest-publish,
  external-query, zenoh-subscribe, …) + a `Grant` model attaching a capability
  to a principal.
- A capability registry (the set of known capabilities).
- `check_capability`: principal + capability → allow/deny, **failing closed**
  (unknown capability and missing grant both deny).
- Grant `create` / `list` / `revoke`, persisted in SurrealDB within the caller's
  own scope — no privilege escalation (a principal cannot grant itself, or
  another, a capability it lacks the authority to grant).
- Load-bearing contracts honored: #2 (capability grants are app-enforced, not
  SurrealDB-native; distinct from the WS-03 row-level layer, same principal).
