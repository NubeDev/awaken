# WS-04 — Edge/cloud profiles (cargo features + runtime config)

Status: Pending
Verified: (re-verify on first action)

## Goal

STACK-DEISGN.md: "single binary, cargo-feature + runtime-config selected." Today there is one build
with all features compiled in. Introduce the profile split (edge vs cloud) as cargo features + a
runtime profile config, so later WSs (Postgres cloud backend, driver runtime) attach to it cleanly.
Ship the plumbing first — this WS is the seam, not the heavy backends.

## Current state (re-grep before coding)

- `rubix/crates/rubix-server/src/main.rs` — boot path; reads all the `RUBIX_*` env toggles, opens the
  store, bus, supervisor, scheduler, agent, dispatch. This is where profile selection slots in.
- `rubix/crates/rubix-server/Cargo.toml` — no feature flags today; add `edge` (default) / `cloud`.
- `rubix/crates/rubix-server/src/lib.rs` — `AppState` and wiring.

## Scope / deliverables

- Cargo features `edge` (default) and `cloud` on `rubix-server`; a runtime `Profile` enum selected by
  `RUBIX_PROFILE` (env) within what the compiled features allow (feature gates capability, runtime
  config selects among compiled options).
- A `Profile` config struct centralizing the per-profile defaults (store kind, history tier, whether
  drivers/supervisor launch, auth-required) — read once at boot, threaded into `AppState`.
- Edge profile preserves today's exact behavior (SQLite, drivers on, no auth) so nothing regresses.
- Cloud profile compiles and boots with the cloud defaults using only what exists today (Postgres
  and auth land in WS-05/WS-06 behind this seam — leave clearly-marked `// TODO(loop):` extension
  points ONLY where a later WS owns the impl, never a fake fallback).
- Tests: profile selection resolves correctly from env+features; edge defaults unchanged.

## Done definition

Green tests+clippy under both `--features edge` and `--features cloud`; edge behavior unchanged;
STATUS.md "Platform / deployment" edge/cloud-profiles item checked; env list updated; log `Done`.
