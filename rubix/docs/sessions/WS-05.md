## Design notes / assumptions

- A record mutation is gated by a `Capability` the `Command` carries (tests use
  `RuleInvoke`). Per the WS-05 spec the gate "checks the capability grant"; the
  enumerated capabilities are cross-plane, so the command supplies which grant a
  given mutation requires rather than the gate inventing a record-write
  capability. The authz call is the WS-04 `check_capability`, unchanged.
- Atomic before/after: `capture` runs the `RETURN BEFORE` statement built by
  `persist` and decodes the before-image; the after-image is the content the
  command wrote (none for delete). One round trip, no read-before-write.
- Audit immutability is engine-enforced (`FOR create, update, delete NONE`); the
  gate appends on the root/owner handle, the only session past that permission.
  Verified on kv-mem: a scoped principal's `UPDATE`/`DELETE` leaves the row
  intact.
- The captured before/after is exposed on `Applied` so WS-06 undo can build the
  inverse from the same capture (one capture, two consumers) with no new read.

## Goal
Build the single write-enforcement point: every mutation crosses the gate, which
authenticates the principal, checks the capability grant, captures before/after
atomically with the write (SurrealDB `RETURN BEFORE`), mints/carries the
correlation id, and only then applies the change (SCOPE "Access & policy gate",
contract #1). Each mutation also produces an append-only, immutable audit row —
who-did-what stamped with the correlation id (SCOPE "Audit log", contract #4).

## Scope / deliverables
- Crate(s): `rubix-gate` (command + audit modules).
- A `Command` abstraction — the unit that crosses the gate (principal, namespace,
  action, target record id, intended change), with a single `apply`-through-gate
  path: authenticate → check capability (WS-04) → mint/carry correlation id →
  atomic before/after capture → apply → write audit row.
- Atomic before/after capture using SurrealDB `RETURN BEFORE` (no separate
  read-before-write round trip), surfacing one captured change consumed by audit
  here and by undo in WS-06.
- An audit record: principal, namespace, action, target record, before/after
  summary, correlation id, timestamp — written append-only to a per-namespace
  `audit` table.
- SurrealDB table permissions forbidding `UPDATE`/`DELETE` on the audit table for
  every principal but the system (immutability enforced by the engine).
- Correlation-id minting at the gate for principal-initiated commands.
- Load-bearing contracts honored (STACK-DEISGN.md): #1 (commands → gate),
  #3 (correlation id minted/carried), #4 (audit append-only + immutable).
