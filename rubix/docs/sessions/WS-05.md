# WS-05 — Postgres backend for the cloud profile

Status: Pending
Verified: (re-verify on first action)
Depends on: WS-04 (profile/feature seam)

## Goal

STACK-DEISGN.md: "Postgres (cloud), SQLite (edge)". Today the store is SQLite-only. Add a Postgres
store implementation behind the WS-04 `cloud` profile, selected at runtime, with SQLite remaining the
edge default. Cloud relational tables (users/teams/config) per STATUS.md "Postgres federation".

## Current state (re-grep before coding)

- `rubix/crates/rubix-server/src/store/mod.rs` — the store API surface (the trait/struct the rest of
  the server calls). Identify the seam to abstract over SQLite vs Postgres.
- `rubix/crates/rubix-server/src/store/schema.rs` — DDL; will need a Postgres dialect variant.
- `rubix/crates/rubix-server/src/store/keyexpr.rs` — keyexpr helpers (store-agnostic, reuse).
- WS-04's `Profile` config — selects store kind. Do NOT add Postgres to the edge default build.

## Scope / deliverables

- A store abstraction (trait) covering the methods the server uses; SQLite impl (existing, refactored
  behind it) + a Postgres impl (sqlx or tokio-postgres + a pool) gated by the `cloud` feature.
- Postgres schema/migrations mirroring the SQLite schema (sites/equips/points/his/sparks/boards/
  widgets + WS-02's runs). Respect the ADR persistence guardrails (A-G2 RunRecord validation, etc.).
- Runtime selection via WS-04's `RUBIX_PROFILE`/`RUBIX_DB` (a `postgres://` URL selects PG under the
  cloud profile).
- Tests: the Postgres impl is covered by the same store test suite as SQLite (gate behind a
  `RUBIX_TEST_PG` URL so CI without a PG instance still passes; do NOT `#[ignore]` to dodge — skip
  cleanly when the env URL is absent and assert when present).

## Done definition

Green tests+clippy on edge (SQLite) and cloud (`--features cloud`); Postgres store passes the shared
store suite when `RUBIX_TEST_PG` is set; STATUS.md Postgres items checked; env list updated; log `Done`.
