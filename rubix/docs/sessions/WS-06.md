## Design notes / assumptions

- **Record kind is caller-declared.** The platform has no fixed ontology — every
  record lives in one generic `record` table — so the undo boundary cannot read a
  table name to classify. `RecordKind` (Definition / DataPlane / Audit) is the
  declared role of the target a mutation acts on; `push_change` admits only a
  `Definition` and fails closed on the other two (`GateError::UndoBoundary`). This
  is the honest classification the SCOPE "definitions only" boundary needs given
  the schemaless model.
- **One capture, two consumers.** `ChangeRecord::from_capture` derives the inverse
  from the *same* `CapturedChange` audit projects from (no new read): create→delete,
  update→update(prior), delete→create(prior).
- **Reversal re-enters the gate.** `undo`/`redo` build a fresh WS-05 `Command` from
  the inverse/forward change and drive it through `apply`, carrying the original
  correlation id — so the reversal is capability-checked and produces its own audit
  row threaded to the same chain (SCOPE "applied through the gate").
- **Stack is in-memory, per principal + resource.** `UndoStore` partitions by
  (subject, namespace, target). Linear undo/redo with redo cleared on a new forward
  change (SCOPE open question 9: per-user linear baseline, collaborative model
  deferred). The store is session-scoped state the transport layer (WS-16) owns; it
  is not persisted, matching the "mutable, short, session-scoped" retention of the
  SCOPE undo table.
- `undo.rs` renamed to `reverse.rs` to avoid clippy `module_inception` (the verb
  file collided with the `undo/` module name); the public function stays `undo`.

## Goal
Make user-facing definition mutations (dashboards, rules, tags, datasource config)
reversible — and only those: never the data plane, never the audit log. Undo
derives from WS-05's captured change (the mutable consumer; audit is the immutable
one) and applies the inverse back through the WS-05 gate, so it is
permission-checked and itself audited (contract #4).

## Dependencies
- **WS-05** — the atomic before/after capture and the `Command::apply` gate
  pipeline. Undo consumes the capture and re-enters the same gate.
