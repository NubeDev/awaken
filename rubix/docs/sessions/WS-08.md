## Goal

Ship the MCP outbound adapter: an MCP server surface exposing the existing BMS tools
(read_point / write_point / query / run_board / pin_widget) to external agents over HTTP, with
identical priority-array gating, tenant scoping (WS-07), and HITL escalation (WS-02 run registry
tracks externally-initiated runs).

## Design

MCP is a tool-call protocol (the external agent does the reasoning; rubix exposes tools), so the
adapter dispatches tool calls directly into `build_tools_scoped` rather than running an inference
loop. Transport is JSON-RPC 2.0 over a single HTTP endpoint (`POST /api/v1/mcp`) — the modern MCP
streamable-HTTP request/response form; no SSE is needed because tool calls are request/response.

A `write_point` that lands in the escalation band suspends (the tool returns a suspended
`ToolResult` with the held write); the adapter persists that as a `RunRecord` of new origin `Mcp` so
the externally-initiated run lands in the same `runs` registry and resumes through the existing
`POST /api/v1/runs/{id}/resume` (origin-agnostic) operator surface. The store is untouched until an
operator approves.

Scope is bound from the request `Principal` (WS-06 bearer middleware): a site-pinned principal
confines the tools to its `{org}/{site}` exactly as the chat endpoint does; a broader/edge request
runs unscoped.

3. Integration tests (`tests/api_tests/mcp.rs`) + a `with_escalation_floor` harness builder.

## Tests

`tests/api_tests/mcp.rs` (8 tests): tools/list advertises the surface with input schemas; a
default-priority call writes through the priority array and read-back confirms; a below-floor write
(floor 5, slot 3) is a hard error result with the point untouched; an escalation-band write (slot 5)
suspends, surfaces `awaiting_approval` + a `run_id`, appears in `GET /runs?status=suspended` as
origin `mcp` with the held write, leaves the point uncommanded, and resumes to commit it; a
site-A-scoped session is refused a site-B write at the tool boundary; unknown method → JSON-RPC
`-32601`; unknown tool → MCP error result (not a transport error); `initialize` advertises the tools
capability. Unit tests in each `mcp` submodule cover the JSON-RPC envelope, param parsing, result
shaping, and pending-write recovery.

## Assumptions / deviations

- The MCP endpoint is raw JSON-RPC, so it is not registered as a typed utoipa path (the body shape
  is method-dependent); the tool schemas it advertises come from the same descriptors the embedded
  agent uses. `RunOrigin`'s new `mcp` value flows into the existing `RunRecord` OpenAPI schema.
- The external session thread id is a fixed marker (`mcp`) since an MCP `tools/call` carries no
  thread; a suspended run is still uniquely keyed by its own run id.
