  child inherits its parent's trace id (contract #3).
- `emit.rs` — `emit_span` publishes a span as a `ControlEvent` of type
  `trace.span` on the WS-07 in-process bus, threaded by its trace id; no-op when
  no subscriber (fire-and-forget, matching the control plane).
- `sample.rs` — `SampleRate` drop-fraction decision from `RUBIX_TRACE_SAMPLE`,
  clamped to `[0,1]`, deterministic per span id (FNV bucket) so the realized rate
  tracks the configured fraction with no shared RNG state.
- `define.rs` — `define_trace_schema` defines the append-only `trace` table
  (`FOR create,update,delete NONE`, scoped `select`), like the audit table.
- `row.rs` — `SpanRow` store projection (adds namespace for row-level scope and a
  store `appended` time for retention ordering).
- `persist.rs` — `persist_span` applies sampling then appends on the root handle;
  returns `Persisted::Written | Dropped`.
- `retain.rs` — `enforce_retention` caps a namespace's spans at `max_spans`,
  evicting the oldest by append time (select victims, delete by id).
- `assemble.rs` — `assemble_trace` reads spans by trace id and links them into
  `SpanNode` trees, orphans (evicted parents) promoted to roots.

## Assumptions / deviations

- The persisted span row carries `namespace` (not on the domain `Span`) so the
  `trace` table's row-level read permission scopes by tenant, consistent with the
  `record`/`audit` tables. `persist_span`/`enforce_retention` take the namespace
  explicitly; the domain span stays tenancy-free.
- Span timestamps are caller-supplied epoch nanoseconds (no clock dependency in
  the model); the store stamps its own `appended` time for retention ordering.
- Sampling is deterministic per span id rather than RNG-based — no shared mutable
  state, stable decision, rate verified statistically in tests.
- Retention "oldest" orders by store `appended` time; under tied timestamps in a
  tight kv-mem loop the cut tiebreaks by id, so the eviction-survivor test asserts
  the bound count rather than specific insert-order survivors.
- `define_trace_schema` is exported for WS-16 to call at bootstrap (the trace
  table is not yet wired into `rubix-server` AppState — that is WS-16's transport
  wiring, like the gate/audit schema today). Not a blocker; no TODOs logged.

## Goal
Answer "how data/decisions flowed": spans emitted onto the event bus as work flows
ingest → pre-process → rule → insight → sink, correlated by the WS-05
trace/correlation id (SCOPE "Tracing", contract #3). Spans persist to SurrealDB
with bounded/rolling retention and a configurable sampling rate — high volume, not
kept forever (contract #4). This WS provides the hook for the Rhai per-evaluation
span tree; WS-11 fills that tree ("why did this fire").

## Dependencies
- **WS-05** — the correlation/trace id minted at the gate that correlates spans.
- **WS-07** — the event bus spans are emitted onto.
- **WS-01** — `rubix-store` for bounded/sampled span persistence.

## Scope / deliverables
- Crate(s): `rubix-trace`.
- A span model: trace id, parent span id, name, attributes, start/end — minimal,
  named by concept (not a `types.rs`).
- Emit spans onto the WS-07 bus as work flows; spans carry the WS-05 correlation
  id as their trace id.
- Persist spans to a SurrealDB `trace` table with bounded/rolling retention and a
  configurable sampling rate, env `RUBIX_TRACE_SAMPLE` (drop fraction before
  persist). Trace writes are append-only.
- A span-tree assembly read path keyed by trace id (assemble parent/child from
  stored spans) — the hook WS-11's Rhai span tree plugs into.
- Load-bearing contracts honored (STACK-DEISGN.md): #3 (correlation id is the
  trace id), #4 (trace append-only, bounded/sampled).
