## Goal
Deliver the unification and compute layer (SCOPE "DataFusion — query and
compute"). DataFusion sits ABOVE SurrealDB only where cross-datasource
unification or heavy vectorized aggregation is wanted (contract #6); SurrealQL
stays first. Provides the read-only SQL surface, vectorized time-window
aggregation that feeds rule decisions, and vector/semantic search over SurrealDB
vector columns.

## Design
DataFusion sits strictly above SurrealDB. The provider scans each canonical
table (`record`/`tag`/`audit`/`insight`) through the principal's scoped
SurrealDB session via plain SurrealQL `SELECT`, projects rows into a fixed Arrow
schema (structural columns + free-form `content` JSON string), and registers them
as in-memory tables in a `SessionContext`. SurrealDB does the permission-scoped
row read (contract #1); DataFusion does the SQL/aggregation (contract #6). There
is no unscoped base table, so a query cannot escape the principal's permissions.

Vector / semantic search runs SurrealQL-first: the k-nearest query uses
SurrealDB's native vector distance on the scoped session, not DataFusion, because
the engine owns the vector index (contract #6 — SurrealQL first).
