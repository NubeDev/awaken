## Design
DataFusion sits strictly above SurrealDB. The provider scans each canonical
table (`record`/`tag`/`audit`/`insight`) through the principal's scoped
SurrealDB session via plain SurrealQL `SELECT`, projects rows into a fixed Arrow
schema (structural columns + free-form `content` JSON string), and registers them
as in-memory tables in a `SessionContext`. SurrealDB does the permission-scoped
row read (contract #1); DataFusion does the SQL/aggregation (contract #6). There
is no unscoped base table, so a query cannot escape the principal's permissions.

Vector / semantic search runs SurrealQL-first: the k-nearest query uses
SurrealDB's native vector distance on the scoped session, not DataFusion, because
the engine owns the vector index (contract #6 â€” SurrealQL first).

## Assumptions / deviations
- The query capability is `Capability::ExternalQuery` (the registered cross-plane
  query grant); WS-09 does not add a capability variant (that is a gate-owned
  shared-file contract change).
- The provider scans through the scoped session and registers in-memory tables
  rather than implementing a streaming async `TableProvider`. This honors the
  STACK-DEISGN contract literally â€” SurrealDB does the permission-scoped row read,
  DataFusion sits strictly above for SQL/aggregation â€” and keeps the scope
  confinement structural (no unscoped base table). A streaming provider is a
  performance follow-up, not a contract change.
- Vector search runs SurrealDB-native (contract #6, SurrealQL first), not in
  DataFusion, because SurrealDB owns the vector data and distance functions.
- The `insight` table is in the canonical set ahead of its WS-11 writer; a scan
  of an absent/empty table returns zero rows (a missing-table engine error is
  treated as empty, never surfaced).
