## Goal
Deliver the embedded, deterministic rule/insight runtime (SCOPE "Rhai — rules and
insights"). Rules are composable (rules call rules), fire offline, consume the
time-window math from DataFusion (WS-09), and produce a decision/insight recorded
back to SurrealDB through the WS-05 gate, published as a WS-07 data-change event,
and accompanied by a WS-08 span tree per evaluation (the deterministic "why did
this fire"). Rhai owns the decision; DataFusion owns the data — heavy aggregation
never lives in Rhai.

emit the per-eval span tree. Consumes the gate command path, the bus publish
seam, and the trace span emitter.

- Composition: a rule invokes another rule.
- Evaluation that pulls DataFusion window values, runs the script, records the

## Decisions / assumptions

- **Insight is recorded through the gate to the generic `record` table.** The
  WS-05 gate write path is hardcoded to the `record` table and is the only
  sanctioned durable write boundary; the spec mandates "records the insight via
  the gate". An insight is an append-only data-plane firing (per
  `rubix-gate`'s undo `RecordKind::DataPlane`), so it is created as a generic
  record whose `content` carries the decision (no fixed ontology, SCOPE
  principle 4). The query crate's separate `insight` canonical table is left for
  a dedicated writer; nothing in WS-11 needed it, and bypassing the gate to
  write it would violate contract #1.
- **Composition is declared, not inferred.** A `Rule` declares `subrules` it may
  `invoke`; the orchestrator pre-evaluates each depth-first and exposes its
  decision value to the parent's synchronous `invoke(id)` host fn. This keeps the
  engine free of async (Rhai is sync) and the dependency graph explicit and
  fail-closed — `invoke` of an undeclared/unevaluated id errors. This adds a
  `subrules` field + `composing()` builder to the proposed `Rule` shape.
- **mod.rs kept a pure barrel** (CLAUDE.md overrides the doc's "mod.rs + Rule"):
  `Rule` lives in `rule/define.rs`, mirroring the gate's `command/define.rs`.
