# WS-13 — Extensions as scoped principals

Status: Pending

## Goal
Model an extension as a service account in SurrealDB auth, scoped to a namespace —
the *same identity model* as a user (SCOPE "Extensions as principals"). Enforcement
is the two layers: SurrealDB row-perms for data records, app-enforced capability
grants for cross-plane actions. The control plane is JSON-RPC (register, configure,
invoke, health, lifecycle); the data plane delegates to WS-12's Zenoh key-space
scoping. Every extension *command* crosses the WS-05 gate, so an extension is
audited identically to a user (contract #1).

## Dependencies
- **WS-03** — `Principal`; an extension is provisioned as a scoped service-account
  principal on the same identity model.
- **WS-04** — capability grants; what an extension may *do* (ingest, invoke a rule,
  register a datasource, subscribe a Zenoh key-space) is a grant on its principal.
- **WS-05** — the command gate + audit; every extension control action crosses it.
- **WS-12** — Zenoh data-plane key-space scoping; the data plane delegates here.

## Scope / deliverables
- Crate(s): `rubix-ext`.
- Provision an extension as a scoped service-account `Principal` (WS-03) bound to a
  namespace — no separate plugin-trust path.
- A JSON-RPC control surface — `register` / `configure` / `invoke` / `health` /
  `lifecycle` — whose actions are capability-gated (WS-04) and routed as commands
  through the WS-05 gate so each writes an audit row.
- Data plane: delegate to WS-12's Zenoh key-space scoping (scope resolved once at
  subscribe, a capability decision), not re-implemented here.
- The same grant mechanism expresses a read-only extension, an ingest-only
  extension, and an admin extension — only the grants differ.
- Load-bearing contracts honored (STACK-DEISGN.md): #1 (extension commands cross
  the gate), #2 (two authz layers — row-perms for data, grants for cross-plane).
