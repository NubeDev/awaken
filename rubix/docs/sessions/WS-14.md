# WS-14 — Edge/cloud profiles (single binary, features + runtime config)

Status: Done

## Goal
The same binary configured two ways (SCOPE "Edge and cloud profiles"). **Edge** =
single SurrealDB namespace, rules fire offline, no multi-tenancy code path (the
gate resolves to the one tenant). **Cloud** = namespace-per-tenant isolation,
teams/users via SurrealDB auth, cloud-only add-ons. Selection is a cargo feature
(`edge` default / `cloud`) plus a `RUBIX_PROFILE` runtime choice among the
compiled-in profiles, centralized in one `Profile` config threaded into `AppState`.

## Dependencies
- **WS-03** — the gate's tenant resolution; on edge it resolves to the one tenant
  automatically, on cloud it resolves namespace-per-tenant.
- **WS-10** — the Postgres connector, a cloud-only backend the cloud profile may
  require; cloud fails closed when it is absent from the build.
- **WS-05** — the gate `AppState` the profile config is threaded into.

## Scope / deliverables
- Crate(s): `rubix-server` (profile module) + cargo features.
- Cargo features `edge` (default) / `cloud`, with `compile_error!` if neither is
  enabled — a profile must always be compiled in.
- `RUBIX_PROFILE` runtime selection among the compiled-in profiles; an unknown or
  uncompiled profile name is rejected at boot with a clear error (no silent
  fallback).
- A `Profile` config centralizing per-profile defaults read once at boot: store
  kind, namespace strategy (single vs. per-tenant), auth required, sync on/off —
  threaded into `AppState`.
- Edge preserves single-namespace behavior; cloud selects namespace-per-tenant and
  **fails closed at boot** when it would need a backend not in the build, with a
  clear boot error (no degraded fallback path).
- Load-bearing contracts honored (STACK-DEISGN.md): SCOPE edge/cloud profiles;
  "No multi-tenancy on edge"; contract #1 (gate tenant resolution per profile).

## Proposed file layout (verb-per-file, ≤400 lines)

