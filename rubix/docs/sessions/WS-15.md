# WS-15 — Edge↔cloud sync shipper over Zenoh

## Goal
SurrealDB has no mature multi-master replication, so sync is an **application-level
shipper over Zenoh**, not DB replication, with an explicit conflict model (SCOPE
"Sync and conflict model"). The **data plane** (readings, insights, audit, traces)
is append-only and edge-owned, partitioned by edge identity, so reconciliation is
ordering + dedup by id, not merge — no multi-master conflict by construction
(contract #5). The **config plane** (dashboards, rules, tags, datasource defs) is
the real conflict surface: ownership first (cloud owns shared/tenant config, edge
owns local-only), then last-write-wins with the WS-05 audit log as tiebreak.

## Dependencies
- **WS-12** — the Zenoh transport the shipper moves records over.
- **WS-02** — the generic record model the shipper carries (data + config).
- **WS-05** — the audit log whose timestamp is the LWW tiebreak on config conflicts.
- **WS-14** — profiles decide who owns what (cloud owns shared, edge owns local).

## Scope / deliverables
- Crate(s): `rubix-sync`.
- A data-plane shipper that moves append-only records from the edge partition
  edge→cloud: ordering + dedup by id, **idempotent replay on reconnect** (a re-sent
  record is a no-op), conflict-free by partitioning on edge identity (contract #5).
- A config-plane reconciler that applies **ownership** first, then **LWW with the
  audit timestamp as tiebreak** only where overlap is unavoidable.
- Clear plane boundaries: data is conflict-free by partition; config is the only
  surface that ever reconciles. No CRDT (deferred per SCOPE open question 1).
- Load-bearing contracts honored (STACK-DEISGN.md): #5 (append-only,
  edge-partitioned, reconcile by ordering + dedup); SCOPE sync/conflict model.
