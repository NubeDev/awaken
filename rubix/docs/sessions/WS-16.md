## Assumptions / deviations
- Principal auth over HTTP uses a `x-rubix-subject` / `x-rubix-secret` header pair
  resolved through `rubix_gate::authenticate` + `issue_scoped_session`, since
  WS-16's spec assumes an authenticated principal but the wire credential shape is
  not separately specified and no JWT/session-cookie WS owns it. This reuses the
  committed `PrincipalToken` pair the gate already verifies natively.

## Follow-ups (logged to TODOs.md 2026-06-15T09:50:00Z)
- **JSON-RPC extension control endpoint** (`rpc/control.rs`) → needs WS-13
  (`rubix-ext`). Not stubbed.
- **`POST /datasources` registration** → needs a `Connector` instance the WS-13
  extension model supplies. `GET /datasources` (list) shipped.
- **Profile selection into `AppState`** → WS-14. The binary boots on the committed
  edge-default `RuntimeConfig`; the transport is profile-agnostic.
- **`insights/list`** — insights are recorded as plain generic records by
  `rubix-rules`; they surface through the scoped record read (`GET /records`). A
  dedicated `insights/list` would need the rules path to tag insights with a
  marker; until then `GET /records` is the read surface. Not a blocker.
- **Record-write capability** — record mutations route through the gate's
  `IngestPublish` capability since the committed `Capability` enum has no dedicated
  record-write variant. If one is added to the gate, change only
  `http/records/capability.rs`.
