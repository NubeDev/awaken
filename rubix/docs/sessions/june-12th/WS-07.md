## Goal

STATUS.md AI layer: "Tenancy: org/site hierarchy mirrored into awaken `ScopeId`." Make the embedded
agent and dispatch runs tenant-scoped — a run carries the org/site it acts within, mapped onto
awaken's tenant scope, so an agent invoked for one tenant cannot read/write another's points even
through its tools.

## Design (as built)

awaken-runtime 0.6 has no general tenancy `ScopeId` and a `TypedTool` gets no per-run scope context.
So a run cannot carry a scope into a shared tool set; instead the scope is bound when the tools are
**constructed**, by building a per-run runtime whose point/board/widget access is wrapped in a tenant
guard. The unscoped boot runtime remains the fallback for runs with no tenant (no sited principal, no
spark site).

- **Mapping (deterministic, documented):** `TenantScope { org, site }` maps to the awaken scope id
  `"{org}/{site}"` — the same first-two-segment prefix of every point keyexpr
  (`{org}/{site}/{equip}/{point}`). `TenantScope::scope_id()` is that string; `covers(keyexpr)` does
  path-boundary matching (equal, or `strip_prefix` then `starts_with('/')`), reused from the driver
  capability model. (`rubix-tools/src/scope.rs`.)
- **Enforcement at the tool boundary:** `ScopedPointAccess` decorates `PointAccess`, denying any
  read/write/his call whose keyexpr is outside the scope **before the store is touched**
  (`rubix-tools/src/scoped/point.rs`). `StoreBoardAccess::scoped` / `StoreWidgetAccess::scoped` wrap
  board runs and widget pins the same way (`rubix-server/src/tools/`). `build_tools_scoped(state,
  Some(scope))` assembles the scoped set. The free-form SQL `query` tool was initially **withheld**
  from scoped runs (fail-closed); a concurrent session (fd37cad0, ba37da2d) added a tenant-filtered
  DataFusion session (`QueryEngine::scoped_query`), so a scoped run now gets a `query` tool confined
  to its own `{org}/{site}`, the withhold kept only as the fail-closed fallback for a scope that
  cannot map to a `QueryScope`.
- **Chat path:** `POST /agent/chat` reads the request `Principal`; when it carries both an org and a
  site, the run is scoped to `{org}/{site}` via `runtime_for_scope` (`api/agent/chat.rs`).
- **Dispatch path:** an inbound spark names its `site_id`; `dispatch_spark` resolves it to the
  owning `{org}/{site}` and builds a scoped runtime, **failing closed** (skipping the run) if the
  site is unknown (`dispatch/run.rs`). Per-run runtimes are rebuilt from a stored `RuntimeBlueprint`
  (`agent/runtime.rs`, `agent/scoped.rs`); `AppState` gained `agent_blueprint`.

## Assumptions / deviations

- Scope is bound at tool construction (per-run scoped runtime), not carried on `RunActivation` — the
  0.6 contract provides no per-run tool scope. The mapping is documented above and in the source.
- The chat path scopes only when the principal carries **both** org and site; a broader principal
  (org-only / unscoped operator) runs with the full tool set, matching the RBAC posture from WS-06.
