## Goal

Rubix has **no SQL macro/variable binder**. Before any dashboard variable, time range, or page
context can drive a query, there must be one server-side, injection-safe interpolation engine that
lowers `$name`/`${name}` tokens into bound parameters. This WS builds exactly that engine and makes
both query entry points accept a `variables` payload. It ships nothing user-facing — it is the
contract WS-02/WS-03/WS-06 consume.

## Current state (re-grep before coding)

- `crates/rubix-query/` — exists today with `context/`, `provider/`, `rollup/`, `sql/`, `his/`,
  `error.rs`, `lib.rs`. It executes queries but has **no variable/interpolation module**. Add the
  engine here so both query paths share one binder.
- `crates/rubix-server/src/api/query/run.rs` — `POST /api/v1/query`, DataFusion over canonical
  tables, request is `{ sql }` only. Has the read-only / timeout / row-cap guards — **reuse them**.
- `crates/rubix-server/src/api/datasources/run.rs` — `POST /api/v1/datasources/{id}/query`, request
  is `{ sql, params: Vec<Value> }` (positional binds, no named interpolation).
- `crates/rubix-server/src/api/openapi.rs` — DTO registration for the new request fields.

## Scope / deliverables

- A new interpolation module in `rubix-query` (verb/concept-named files, ≤400 lines each):
  - `QueryVariable { name, value | values }` input type.
  - Lowering of, in SQL text, **before** execution:
    - `$name` / `${name}` → single value bound as a positional/`$N` parameter.
    - `${name:csv}` → comma-joined, **each value its own bound parameter**.
    - `${name:singlequote}` → quoted list for `IN`-style use, each value a bound param.
    - `$__sqlIn(name)` → safe `IN ($1, $2, …)` expansion from a multi-value variable.
  - An unknown/unbound token is a clear engine error, not a silent passthrough.
- **The injection boundary:** every value becomes a bound parameter, never concatenated. A value of
  `'); DROP TABLE points; --` binds as a literal. Quoting/escaping lives in the engine, server-side,
  mandatory. Do **not** invent a second read-only/timeout/row-cap guard — reuse the ones already on
  the two query paths.
- Extend `QueryRequest` and `DatasourceQueryRequest` with `variables: Vec<QueryVariable>` (default
  empty → existing behaviour unchanged, back-compat). Register the new field shape in `api/openapi.rs`.
- Wire both `run.rs` handlers to lower the SQL through the engine before execution.

## Done definition

Green `cargo test --workspace` + clippy clean; engine unit tests cover each substitution form,
multi-value expansion, and an injection attempt that binds-not-executes; both query paths accept and
apply `variables` with no payload meaning no change; OpenAPI surface compiles; session log `Done`.

## Notes / boundaries
- This WS does **not** add the `variables` field to the dashboard model or any UI — that is WS-02.
- It does **not** add the time macros (`$__from`/`$__timeFilter`/…) — those extend this engine in
  WS-03, but ship the engine so that extension is "add one token handler", not a rewrite.
