## Goal

Supply the `$__from`/`$__to`/`$__interval` built-ins the variables doc references and let a global
`{from, to, interval}` flow into every time-aware widget query — server-side, injection-safe. This is
the **backend half** of time-range; WS-04 is the UI half.

## Scope / deliverables

- Extend the WS-01 engine with time substitutions, every value **bound**, never inlined:
  - `$__from` / `$__to` → resolved range bounds (bound parameters).
  - `$__timeFilter(col)` → `col >= $from AND col < $to`.
  - `$__timeGroup(col, '$__interval')` → bucketing by the resolved interval.
- Extend `QueryRequest` + `DatasourceQueryRequest` with `time_range: Option<{from, to}>` and
  `interval_secs: Option<u32>` (register in `api/openapi.rs`). A widget whose SQL uses no time macro
  is unaffected (back-compat).
- **Freeze one `now` per refresh, server-authoritative:** resolve `now` once per request set and pass
  the resolved instant to every macro so a fan-out of widgets shares one instant (no per-widget skew).
- **`$__interval` auto-calc:** derive a bucket from `(to - from) / targetDataPoints`. Decide with the
  engine owner whether the client passes `interval_secs` or the engine computes; spec leaves room for
  either — but the *resolved* interval is what binds.
- **`point_history` wiring:** thread the resolved range into the existing `his` `start`/`end` params
  rather than fetch-all-and-slice (the UI side of the fetch is WS-04; here ensure the his query path
  accepts and honours the resolved bounds).

- Commit 2 (server/DTO/TS): `QueryRequest` + `DatasourceQueryRequest` carry `time_range` +
  `interval_secs`; `api/time_range.rs` freezes one server `now` per request and resolves the range
  (a bad token / empty range is a 400). `TimeRangeBody` registered in OpenAPI; TS mirror
  (`types.ts` + `endpoints.ts` `query.run` opts) extended additively.

## Decisions / assumptions
- **Interval:** the client may pass `interval_secs`; when absent (or 0) the server derives a bucket from
  `(to - from) / 200`, clamped to >= 1s. The *resolved* width is always what binds (per spec, "the
  resolved interval is what binds"). 200 is a server-side fallback; WS-04 can pass a pixel-accurate width.
- **`$__timeGroup` SQL:** `FLOOR(EXTRACT(EPOCH FROM CAST(col AS TIMESTAMP)) / $N) * $N` — portable across
  DataFusion and Timescale; only the operator-authored column splices (same trust tier as the widget SQL),
  the width binds.
- **`point_history`:** the his path (`GET /points/{id}/his?start=&end=`) already binds half-open
  `>= start AND < end` (matching `$__timeFilter`), so no backend change was needed; threading the resolved
  range into these params is the WS-04 UI fetch.
