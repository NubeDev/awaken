## Goal

Supply the `$__from`/`$__to`/`$__interval` built-ins the variables doc references and let a global
`{from, to, interval}` flow into every time-aware widget query ‚Äî server-side, injection-safe. This is
the **backend half** of time-range; WS-04 is the UI half.

## Current state (re-grep before coding)

- `crates/rubix-server/src/api/his/query.rs` ‚Äî `GET /api/v1/points/{id}/his?start=&end=&limit=`,
  `HisQuery { start, end, limit }`. The only place rubix has a time bound today.
- `crates/rubix-server/src/api/query/run.rs` ‚Äî `POST /api/v1/query` `{ sql }`.
- `crates/rubix-server/src/api/datasources/run.rs` ‚Äî `{ sql, params }`.
- The WS-01 interpolation module in `rubix-query` ‚Äî extend it with the time-macro token handlers.

## Scope / deliverables

- Extend the WS-01 engine with time substitutions, every value **bound**, never inlined:
  - `$__from` / `$__to` ‚Üí resolved range bounds (bound parameters).
  - `$__timeFilter(col)` ‚Üí `col >= $from AND col < $to`.
  - `$__timeGroup(col, '$__interval')` ‚Üí bucketing by the resolved interval.
- Extend `QueryRequest` + `DatasourceQueryRequest` with `time_range: Option<{from, to}>` and
  `interval_secs: Option<u32>` (register in `api/openapi.rs`). A widget whose SQL uses no time macro
  is unaffected (back-compat).
- **Freeze one `now` per refresh, server-authoritative:** resolve `now` once per request set and pass
  the resolved instant to every macro so a fan-out of widgets shares one instant (no per-widget skew).
- **`$__interval` auto-calc:** derive a bucket from `(to - from) / targetDataPoints`. Decide with the
  engine owner whether the client passes `interval_secs` or the engine computes; spec leaves room for
  either ‚Äî but the *resolved* interval is what binds.
- **`point_history` wiring:** thread the resolved range into the existing `his` `start`/`end` params
  rather than fetch-all-and-slice (the UI side of the fetch is WS-04; here ensure the his query path
  accepts and honours the resolved bounds).
