## Goal

The page-context **frontend + the `context` variable source**: a page reads its URL / nav-node
position / tags and feeds those into variables, so one board rendered under Building-1 queries
Building-1's data and under Building-2 queries Building-2's — no second board.

## Scope / deliverables

- **`context` VariableKind (full-stack):** Rust DTO enum arm + config struct in `model.rs`, OpenAPI
  registration, TS union + parser + `resolve.ts` arm + editor form, export/import, tests at each
  layer. Config `{ source: 'nav'|'url'|'tag'|'values', key }`. Built-in tokens (no authoring):
  `$__nav_slug`, `$__nav_name`, `$__tag(key)`. A `query` variable can reference a `context` variable
  as a parent — rides the same injection-safe binder, no new interpolation.
- **`PageContext` assembly** (`ui/src/features/variables/context.ts`, new): assemble from sources kept
  **separate** — `nav` (node_id/slug/name/path), `url` (bare params), `tags` (board tags merged under
  `NavNode.context.tags`), `scope` (existing org/site), `values` (`NavNode.context.values`).
  **Precedence (later wins) per name:** board tags → `nav.values` → URL params → explicit var-bar
  selection. `NavNode.context` is exactly `{ values?, tags? }` — not arbitrary keys. Context is
  read-only input to resolution, **never a 4th store / no `varOverrides` channel**.
- **Resolution wiring:** fold resolved context **into the variable values** that drive `varRevision`
  so a context change bumps the one revision like any selection; **re-key variable resolution** on the
  assembled context too (nav/tag/url) so two mounts of the same board don't serve each other's cached
  option lists. Reuse WS-02/WS-04's dependency-driven invalidation; do not add a parallel one.
- **`TagEditor`** mounted on the board (`kind="dashboard"`); tags feed `PageContext.tags`.
- **Nested sidebar** (`ui/src/features/nav/**`): collapsible tree replacing the flat list; `group`
  nodes expand/collapse, `dashboard`/`route` nodes are links; sidebar shows only nodes the user holds
  `view` on (server-filtered by WS-05).
- **Navigation builder + routing:** add/nest/reorder nodes, pick a target (board + context, or static
  route), manage grants inline. A `dashboard` node opens `…/dashboards/:slug?nav=:nodeId` → read
  `nav` → load node → merge `node.context` into `PageContext`; a `route` node navigates to the static
  page. Bare URL params (`?building=b1`) drive the `url` source for external deep-links — keep the
  `var-` prefix reserved (don't collide).
