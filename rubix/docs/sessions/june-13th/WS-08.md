## Goal

Make the ledger "for everything": drop a `recorder.record(...)` call into every mutation handler
inside its transaction, redact secrets at record time, attribute agent edits to `Actor::Agent`, and
ship the audit-read + undo/redo HTTP surface. This is where "audit and undo for everything" becomes real.

## Scope / deliverables

- **Record on mutate** for every kind in the doc's table (site, equip, point-config, dashboard,
  widget, board, rule, datasource, user/team, grant, token, org), each in its own handler, **inside
  the mutation's transaction**, using the already-read `before` + the written `after` (no extra fetch).
  `group_id` per logical operation.
- **Cascade:** the site delete handler captures each child snapshot before the cascade and records one
  group (shared `group_id`) so undo restores the whole subtree atomically.
- **Secret redaction (recording contract):** `before`/`after` must **never** capture a plaintext
  secret — datasource connection strings/passwords, PATs, agent API keys. Redact those fields at
  record time (`"***"` sentinel). Owners of secret-bearing kinds assert this in tests.
- **Agent actor:** the agent runtime's edits record with `Actor::Agent{run_id, model}` — undoable by
  the user.
- **Audit read surface** (org-scoped, **capability-gated** behind admin/`audit:read`):
  `GET /api/v1/audit` (filter by kind/resource_id/actor/op/time range; paged, newest-first; returns
  `before`/`after`), `GET /api/v1/audit/{kind}/{id}` (one resource's timeline). Never leak another
  org's rows (org filter + grant check). DTOs derive `ToSchema`, register in `api/openapi.rs`; TS
  mirror authored.
- **Undo/Redo:** `POST /api/v1/undo` / `POST /api/v1/redo` — per authenticated principal, pop/advance
  the **per-actor** cursor (CAS-guarded against double-pop), dispatch the group through the reverser
  registry, return the affected `group_id` + touched resource ids (so the UI invalidates exactly those).
  Undo targets **your** changes only — cross-actor global undo is a non-goal.

## Notes / boundaries
- Variable/context edits ride the **dashboard** snapshot (WS-02/WS-06) — recorded once under the
  `dashboard` kind, **no double-record**. Nav/tag mutations (WS-05/WS-06) record here under their kinds.
- The UI (undo/redo shortcuts, toast, History tab, admin audit screen) is **WS-09**.
- Full identity (user/team) undo beyond snapshot restore is a fast-follow; audit first.
