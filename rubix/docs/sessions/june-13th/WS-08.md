## Assumptions / deviations

- **Recording atomicity:** WS-07's `record_in_sqlite_tx` requires a caller-owned `rusqlite::Transaction`,
  but the store's domain mutators (`create_dashboard`, …) each open their *own* internal connection/tx and
  expose no transaction-threaded variant. True single-tx atomicity per the design ("inside the same DB
  transaction") would require refactoring every mutator across all kinds to accept a `&Transaction` — a
  large change spanning shared store files for ~12 kinds, outside one session's safe lane. This WS records
  via the public `record_change` immediately after the successful mutation (still the sole substrate write
  path, still validated). On the SQLite/Postgres happy path the two writes are sequential, not atomic; a
  crash between them could leave a mutation un-recorded. Logged to TODOs.md as the transaction-threading
  follow-up.

- **Kind coverage:** the registry ships only the `dashboard` reverser (WS-07). Wiring the remaining kinds
  (equip/point/widget/board/rule/datasource/user/team/grant/token/org) needs (a) a reverser per kind and
  (b) a `record_change` call in each handler. This session wires the proven `dashboard` kind end-to-end
  and ships the full audit/undo HTTP surface; the remaining per-kind reversers + handler calls + the
  agent-actor plumbing (`StoreWidgetAccess` carries no run_id/model today) are logged to TODOs.md.
