## Goal

The audit/undo **frontend**: undo/redo control + shortcuts that refresh exactly the touched queries,
a toast with inline undo after a mutation, a per-resource History tab (timeline + before→after diff),
and an admin Audit log screen gated behind the audit capability.

## Assumptions / deviations
- **Touched-id → kind mapping.** `UndoResult.touched` carries resource *ids* only (no kind), and one
  undo group can span kinds (widgets + dashboard layout). Rather than guess a kind per id, `useUndo`/
  `useRedo` invalidate the fixed set of config-entity key roots (`widgets`, `dashboards`, `points`,
  …) when `touched` is non-empty, then invalidate `['audit']`. TanStack dedupes; this refreshes the
  canvas exactly as the design intends ("refreshes without a reload") without a per-id kind lookup the
  server doesn't expose.
- **Undo is server-driven by the actor cursor**, so the toolbar buttons are always enabled when an org
  is in scope (no client-side "can undo?" prediction); an at-the-end pop returns an empty group and we
  toast "Nothing to undo". This avoids a client change-log, per the WS boundary.
- **History tab surface.** Mounted as a `HistoryButton` dialog (reusable on any resource detail view)
  rather than restructuring the builder into a tab strip — least-invasive, and the design says "a
  History *tab* on a dashboard, datasource, rule …" which the dialog satisfies for the dashboard now;
  the same `HistoryButton`/`HistoryTab` drop onto rule/datasource detail views when those gain a
  toolbar (no new code).
- **Capability gate.** The server gates `/audit` on `require_admin(org)`. The UI mirrors this with the
  existing `AdminGuard` (`whoami.can_admin`) on the screen + `ADMIN_NAV_TITLES` hiding the sidebar item
  for non-admins. The History *timeline* route is likewise admin-gated server-side; the `HistoryButton`
  is shown to any operator but a non-admin sees the 403 surfaced as the tab's error state (no leak).
