## Goal

The look the operator approved is the *populated* look. Before the demo layer can be deleted
(UI-03), the same building portfolio must exist as **real rows in the real store**, served by the
real endpoints, with live `cur` updates — so the UI stays gorgeous with zero fake data in it.
This is the only UI-queue row allowed to touch the backend.

## What to build

1. **Seed path** (backend): a dev-gated way to populate a fresh store with the demo portfolio.
   Preferred shape: a `seed` subcommand or `--seed-dev` flag on `rubix-server` (or a small
   `rubix-seed` bin if the server binary must stay lean — check existing CLI parsing in
   `rubix-server/src/main.rs` and follow its conventions). It must write through the **store
   layer** (or HTTP API), never raw SQL, so every invariant (priority array validation, history
   on effective-value change) holds.
2. **Portfolio content** — port from `ui/src/api/demo/fixtures.ts` (then UI-03 deletes it):
   - 4 sites (hq-tower, distribution-w, lab-campus, cold-store-3), org `acme`.
   - Per site: the 9-equip blueprint (AHUs, chillers, boiler, main incomer, VAVs, tower) with
     Haystack-ish tags (`ahu`, `chiller`, `elec meter`, `submeter`, `comfort`, …) — the dashboard
     derives Load Breakdown from `submeter` tags and KPIs from `meter`/`comfort` tags, so tags
     are load-bearing.
   - Per equip: the point blueprint incl. AHU-3's command points with populated priority arrays
     (slot 13 agent, slot 16 schedule, etc.), the kW total + per-system submeters, comfort index.
   - **History backfill**: 7 days of 30-min samples per numeric point (deterministic curves are
     fine — they're real rows once written through `POST /points/{id}/cur` / the his store).
   - The 7 sparks per site (`POST /sparks` so zenoh publication fires like production).
3. **Live values**: wire `rubix-driver-sim` (already supervised — see WS-09) so seeded sensor
   points tick in dev. If granting the sim all seeded sites is disproportionate, a minimal
   interval task in the seed path that POSTs fresh `cur` values is acceptable — through the real
   ingest endpoint only.
4. **Reference screenshots**: with server+seed running and the UI still in demo mode OFF
   (`VITE_DEMO=0`), capture dashboard/points/sparks/flows via a committed
   `ui/scripts/screenshot.mjs` (playwright, already a devDep) into `ui/docs/reference/`.
   These are the look-freeze baseline for every later gate.
5. Document the dev loop in `ui/README.md`: server + seed + UI in three commands.

## Constraints

- Seed is **dev-gated** (flag/subcommand, never default) and idempotent (re-running must not
  duplicate; upsert by slug).
- Backend gate applies: `cargo test --workspace` + clippy clean. File-layout discipline
  (verb-per-file, ≤400 lines) for any new Rust.

## Done means

- Fresh checkout → `cargo run -p rubix-server -- --seed-dev` (or documented equivalent) →
  `VITE_DEMO=0 pnpm dev` → all four surfaces render populated, matching the committed reference
  screenshots.

## Session log

### First action — verify cited claims
- `rubix-server/src/main.rs` CLI parsing: env-driven only (`env_or`), no clap; a `--seed-dev`
  flag is added by inspecting `std::env::args()` (matching the env-gated convention).
- Store write path confirmed: `Store::create_site/create_equip/create_point/command_point/
  ingest_cur/his_insert/create_spark` enforce invariants (priority-array set/validate,
  history-on-change). Seed writes through these, never raw SQL.
- `Point` `cur`/priority slots cannot be set via `POST /points` (handler only seeds
  `relinquish_default`); the store layer is the correct seam for populated priority slots, so the
  seed module calls `command_point` per slot and `ingest_cur` per sensor sample.
- Spark `acknowledged` is forced `false` by the HTTP handler; seed writes sparks (incl.
  acknowledged ones) through `Store::create_spark`.
- `validate_slug`: lowercase/digit/hyphen only; tag keys: alphanumeric/`_` only. Blueprint slugs
  and tag tokens comply.
