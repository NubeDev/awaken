# UI-05 — Agent surface: runs, resume/cancel, diagnose, HITL UX

Status: Pending
Depends: UI-03 (UI-01 fixed the run types)

## Goal

The agent is rubix's differentiator and the UI barely shows it. Make Agent Runs a real surface
and complete the diagnose→approve loop the backend already supports (WS-01/WS-02 shipped
`agent_call`, the persistent run registry, and `/runs/{id}/resume`).

## Wire facts (re-verify against OpenAPI)

- `GET /api/v1/runs`, `GET /api/v1/runs/{id}`, `POST /api/v1/runs/{id}/resume`,
  `POST /api/v1/runs/{id}/cancel` (`rubix-server/src/api/runs/`).
- `POST /api/v1/agent/chat` → `{ response, steps, status: completed|awaiting_approval, run_id? }`.
- Agent writes land at priority ≥ `RUBIX_AI_MIN_PRIORITY` (403 below) — the priority array's
  level-13 "awaken agent" slot is the real trace of agent action.

## Tasks

1. **Runs list** (`features/runs/`): real columns from the UI-01-verified run shape — id, status
   badge (`awaiting_approval` = warning variant), origin/title if present, started/finished ago.
   Poll while any run is non-terminal.
2. **Run detail**: route `/runs/$runId`; render whatever the record exposes (status, timestamps,
   step/tool trace if the DTO carries it — UI-01 knows). `Resume` / `Cancel` buttons → the real
   endpoints, disabled unless status allows them. If the detail DTO is too thin to render a
   trace, show what exists and file the gap in `../TODOs.md` (backend owns trace exposure).
3. **Diagnose loop** (`features/sparks/components/spark-detail.tsx`): already POSTs chat. Extend:
   when the reply is `awaiting_approval` with `run_id`, the toast gains "Review & approve" →
   navigates to `/runs/$runId` where Resume completes the loop. After resume, invalidate points
   queries so the priority array shows the agent's write at its level.
4. **Approval visibility**: sidebar "Agent Runs" nav badge = count of `awaiting_approval` runs
   (same pattern as the Sparks badge in `app-sidebar.tsx`).
5. Command palette: add "Ask awaken…" action that opens a minimal chat sheet posting to
   `/agent/chat` on a persistent `thread_id` (localStorage), rendering the final response.
   Plain request/response — no streaming (the endpoint is synchronous).

## Out of scope

Full chat transcript UI with tool-call rendering — follow-up once the run record exposes a
structured trace.

## Done means

A spark can be diagnosed; a gated write produces an awaiting-approval run visible in the nav
badge; Resume executes it; the point's priority array visibly gains the agent slot. All against
the real server.
