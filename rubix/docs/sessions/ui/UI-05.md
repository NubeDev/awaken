## Wire facts (re-verified against live `/api-docs/openapi.json`)

- `GET /api/v1/runs` ‚Üí `[RunRecord]` (optional `?status=` filter).
- `GET /api/v1/runs/{id}` ‚Üí `RunRecord` (404).
- `POST /api/v1/runs/{id}/resume` ‚Üí **`ResumeResponse { run_id, point, priority, effective? }`**
  (NOT a `RunRecord`; 400/403/404/409). The UI-01 client typed this as `RunRecord` ‚Äî corrected.
- `POST /api/v1/runs/{id}/cancel` ‚Üí **204 No Content** (NOT a `RunRecord`; 404/409). Corrected.
- `POST /api/v1/agent/chat` ‚Üí `ChatResponse { response, steps, status, run_id? }` where
  `status ‚àà {completed, awaiting_approval}` (this is `ChatStatus`, distinct from `RunStatus`).
- `RunStatus ‚àà {completed, suspended, resumed, cancelled}`. **`suspended` is the
  awaiting-approval state for a run** (the spec's "Wire facts" line conflated this with the
  chat-response `awaiting_approval`; UI-01's `types.ts` is the correct shape). Only a
  `suspended` run can be resumed/cancelled (409 otherwise).
- Resume/cancel re-check the escalation floor server-side; the UI only gates the buttons on
  `status === 'suspended'` and surfaces server errors.
