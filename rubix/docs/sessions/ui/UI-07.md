## Test-coverage note

Per-page render tests were NOT added to the jsdom unit suite: it has no React Testing
Library and the pages depend on TanStack Router `getRouteApi`/`useSearch`, which the existing
render tests exercise only in the `vitest-browser-react` browser suite. Adding RTL is new test
infra outside this row's scope; page-render coverage stays in the browser suite (survivors
fixed to the Rubix nav). The unit suite covers the auth header, 401 handling, the token store,
and the existing logic/contract tests (34 green).

## Assumptions / deviations / follow-ups

- DRIFT vs spec: the spec calls Display a real provider page ("appearance/display drive the
  real theme/font/layout providers"). Display actually submits a fake "Sidebar items"
  checkbox set to `showSubmittedData` with no provider effect; only Appearance drives real
  providers. Kept Display anyway (spec names it a keeper; look-freeze forbids losing pages),
  but it remains UI-only confirmation, not wired state.
- DRIFT: `profile-dropdown.tsx` was already de-fictionalised by an earlier row (no
  "satnaing"); only dead menu items remained, now removed.
- DECISION: no hard `beforeLoad` auth wall on `_authenticated`. The token model has no
  configured issuer and the server accepts dev/seed reads; gating every route behind a token
  would regress the look-freeze to a sign-in wall. 401-driven redirect (per spec) is the
  minimal correct behaviour. Full OIDC redirect flow is a deployment follow-up.
- axios, `lib/utils` `sleep`/`getDisplayNameInitials`, and assorted template components
  (data-table/*, top-nav, etc.) remain knip-unused but are out of this row's stated scope.
- Look-freeze: not re-captured this session (no live server+seed run). Changes are auth
  chrome + residue deletion; no populated dashboard/points/sparks/flows page lost widgets,
  density, or chart series.
